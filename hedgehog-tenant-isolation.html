HEDGEHOG & TENANT ISOLATION

How a Kubernetes YAML file becomes a hardware-enforced private network — multi-tenancy for bare-metal clouds, where you can't trust a single byte of software on the tenant's server.

1

What it is — two ideas bolted together

Control plane / SoftwareONF-CTL

Kubernetes, but the "nodes" are switches.

Hedgehog Open Network Fabric is an open-source control plane where every piece of the network is a Kubernetes object: switches, cables, tenants, subnets. You describe the network you want in YAML (CRDs); a Fabric Controller computes the config each switch needs; a Fabric Agent on every switch applies it and keeps it applied.

This is the same pattern Kubernetes uses for pods, and the same pattern the Metal³ project uses for servers (a BareMetalHost resource per machine) — here the resource is a Switch, and "provisioning" means rendering a fabric config instead of installing an OS.

Tenancy model / HardwareONF-VPC

AWS-style VPCs, enforced by switch silicon.

The tenant-facing abstraction is the VPC: an isolated private network with its own subnets, VLANs, and optional DHCP — deliberately shaped like a public-cloud VPC. Underneath, each VPC compiles to a VRF (a separate routing table) on the switches, and each subnet to a VXLAN VNI.

Isolation is the default: tenants cannot reach each other unless you explicitly create a peering. And because it's enforced in the switch ASIC — not in any software on the server — it holds even when the tenant has root on the entire machine.

The relationship in one sentence: the CRD is the intent, the VRF is the enforcement. You write "tenant-a gets 10.0.1.0/24 on these ports"; Hedgehog turns that sentence into silicon-level separation on every switch in the path.

2

The job it solves: tenancy without a hypervisor

In a public cloud, the hypervisor is the isolation boundary — security groups and virtual switches run in software underneath every guest VM. In a bare-metal GPU cloud, that layer doesn't exist: the tenant owns the whole server, root included, NIC included. Every isolation mechanism that lives on the host is one the tenant can turn off. The only trustworthy enforcement points left are the ones you own: the switches (and, at finer grain, DPUs).

Tenant has rootHost firewalls, OVS, iptables — all tenant-controlled, all worthless as boundaries. The switch port is your first trusted device.
Everyone wants 10.0.0.0/16Every customer arrives with the same private ranges. VRF-per-VPC lets identical addresses coexist on shared switches.
Tenants churn weeklyGPU renters come and go. Onboarding must be an API call from your portal/billing system, not a switch-CLI change window.
RDMA hates overlaysHost-software encapsulation adds latency GPU traffic can't afford. VXLAN in the switch ASIC costs (nearly) nothing.
Audit asks "prove it"Intent lives in Git as YAML; the controller reconciles switches to it. Drift is corrected, and the diff is reviewable.
Customers expect AWSThey know VPC / subnet / peering semantics. Speaking that language cuts your support and product surface dramatically.
The core promise: the isolation boundary moves from the hypervisor into the fabric. Tenants share racks, switches, and wires — but never a routing table. And the whole thing is driven by kubectl apply.

What the fabric actually gives you

  • VPCPer-tenant private network: multiple named subnets, user-defined VLANs, optional per-subnet DHCP (with PXE boot URL support).
  • ATTACHMENTVPCAttachment binds a VPC subnet to a server port (a Connection) — this is the moment a physical machine joins a tenant.
  • PEERINGVPCPeering opens controlled routes between two VPCs — e.g. many tenants reaching one shared storage VPC.
  • FINE ISOLATIONPer-subnet isolated flag (no inter-subnet talk inside a VPC), restricted flag (hosts in a subnet can't even see each other), plus permit lists for exceptions.
  • EXTERNALExternal / ExternalPeering connect chosen VPC subnets to the outside world via BGP on a border leaf.
  • LIFECYCLEDay-0 ZTP switch bring-up via ONIE, Day-1 topology onboarding, Day-2 continuous reconciliation and telemetry (Grafana Alloy → Prometheus/Loki).
  • GUARDRAILSIPv4Namespace / VLANNamespace resources guarantee non-overlapping address and VLAN allocations across the fabric.

From YAML to silicon — the isolation path

Follow one tenant top-to-bottom: a VPC object applied to the Kubernetes API is compiled by the Fabric Controller into a per-switch Agent spec; the on-switch Fabric Agent renders it into SONiC config; the ASIC ends up holding a VRF whose routes only that tenant's ports can reach. Two tenants on the same leaf switch live in different routing tables — same wires, zero shared forwarding state.

intent / control flow tenant A (VRF + VXLAN) tenant B (VRF + VXLAN) ASIC — enforcement
FIG. 3-1 — One leaf, two tenants. The port's VLAN tag selects the VRF; the VRF selects the VXLAN VNI; EVPN advertises routes fabric-wide, but only within the tenant's own VRF. Nothing on the servers is trusted or needed.
The signature trick: classification happens at the first port you own. From that point on, the tenant's packets are stamped into their VNI in hardware, and no configuration the tenant can perform on their server moves them into someone else's network.

Hands-on — the four objects that define a tenant

These are real resource shapes from the Hedgehog docs (trimmed for clarity). Notice there is no switch CLI anywhere: tenancy is created, attached, opened up, and audited entirely through the Kubernetes API.

# The tenant. Isolation is structural: no route to any other VPC exists.
apiVersion: vpc.githedgehog.com/v1beta1
kind: VPC
metadata:
  name: tenant-a
spec:
  ipv4Namespace: default   # guarantees non-overlapping ranges
  vlanNamespace: default   # guarantees non-overlapping VLANs
  defaultIsolated: true    # subnets can't talk to each other…
  defaultRestricted: true  # …and hosts can't talk within a subnet
  subnets:
    gpu:
      subnet: 10.0.1.0/24
      gateway: 10.0.1.1
      vlan: 1001
      dhcp: { enable: true }   # fabric-served DHCP, PXE-capable
Takeaway: the strictest posture is two booleans. isolated = subnets sealed from each other inside the VPC; restricted = hosts sealed from each other inside a subnet. Permit lists then re-open only what you name.
# The moment a physical server joins the tenant: bind a VPC subnet
# to a Connection (a wiring-diagram object naming real switch ports).
apiVersion: vpc.githedgehog.com/v1beta1
kind: VPCAttachment
metadata:
  name: tenant-a--gpu--server-01
spec:
  subnet: tenant-a/gpu
  connection: server-01--mclag--leaf-01--leaf-02

# Or, the CLI shorthand on the control node:
kubectl fabric vpc attach --vpc-subnet tenant-a/gpu \
  --connection server-01--mclag--leaf-01--leaf-02
Takeaway: attachment references the Connection from the wiring diagram — the same object that documents the physical cable. Intent and physical truth share one source, so "which tenant is on this port?" is always answerable with kubectl get.
# Isolation is default; sharing is explicit. Classic use: every tenant
# peers to one shared-services VPC (storage, model registry).
apiVersion: vpc.githedgehog.com/v1beta1
kind: VPCPeering
metadata:
  name: tenant-a--shared-storage
spec:
  permit:
    - tenant-a: {}
      shared-storage: {}

# External world: bind a subnet to upstream BGP on a border leaf.
kind: ExternalPeering
spec:
  permit:
    vpc: { name: tenant-a, subnets: [gpu] }
    external: { name: upstream-isp, prefixes: [{ prefix: 0.0.0.0/0 }] }
Takeaway: peering imports routes between two VRFs — nothing more. Subnet-level permit lists scope it down further, and internet access is just a peering to an External on the border leaf. Every opening is a named, versioned object you can find in Git.
# What the Fabric Agent renders into SONiC (conceptual, simplified):
VRF        VrfVtenant-a                 # one routing table per VPC
VLAN       1001 → VNI 100101           # subnet → VXLAN segment
VTEP       leaf-01 loopback             # tunnel endpoint per leaf
BGP EVPN   Type-2 (MAC/IP) + Type-5 (prefix) routes,
           RD/RT allocated deterministically per VPC+subnet
PORT       Ethernet1 access VLAN 1001   # classification at first hop

# Verify from the control node — status flows back the same channel:
kubectl get vpc,vpcattachment,vpcpeering
kubectl get agents               # per-switch applied generation / heartbeat
Takeaway: the compile target is standard EVPN/VXLAN — the same design hyperscalers run — configured via gNMI/SONiC APIs. If Hedgehog vanished tomorrow, the on-switch state is readable, standard networking.

Alternatives — honest comparison

Fabric-level multi-tenancy for bare metal has four realistic paths. Hedgehog does not win every row.

Lock-in / costNo per-port licenses on the open coreSubscription; support includedHardware + license couplingNo license — but years of engineering
Battle scarsYounger project35+ AI clusters in production, large neocloudsDecades in enterprise DCsDepends entirely on your team
  • PICK HEDGEHOG WHEN you want an open, GitOps-native fabric on white-box SONiC switches, your operators live in Kubernetes, and you want to study or extend the source.
  • PICK NETRIS WHEN you're running a revenue-bearing GPU cloud today and need InfiniBand/NVLink/DPU tenancy plus a mature tenant edge (NAT, elastic IPs, load balancing) with a vendor on the hook.
  • PICK A VENDOR MANAGER WHEN you're standardized on one switch vendor already and multi-tenancy needs are modest.
  • PICK DIY WHEN the fabric is your product differentiator and you can staff protocol-level networking engineers permanently. (Most teams underestimate this by 2–3×.)

Where this pattern shows up

Neocloud GPU rental

Each customer's cluster is a VPC; the portal creates VPC + VPCAttachment objects at checkout and deletes them at churn. Billing meters per-VPC objects.

kubectl · Terraform · GitOps (Flux/Argo CD)

AI factory, per-workload isolation

One organization, many teams/jobs on shared GPUs: a VPC per workload keeps RDMA storage traffic, training traffic, and inference traffic in separate hardware domains.

Hedgehog VPC · SONiC · Spectrum-X

Bare-metal-as-a-service

Pair with server provisioning (Metal³ / Ironic / MAAS): the provisioner installs the OS, a VPCAttachment drops the finished machine into the customer's network. Fabric DHCP can even PXE-boot the installer.

Metal³ · Ironic · pxeURL

Cloud-platform backend

OpenStack or a custom cloud controller emits fabric intent: tenant created in the platform → VPC created in the fabric, port bindings → attachments. The fabric becomes an implementation detail behind your own API.

OpenStack Neutron · REST/K8s API · admission webhooks

Ecosystem — names you'll meet, decoded

Inside Hedgehog

  • Fabric ControllerThe reconciler on the control node: CRDs in, per-switch desired state out.
  • Fabric AgentRuns on every switch; watches its Agent CRD, renders SONiC config, reports status.
  • Fabricator / hhfabInstaller CLI: builds control-node images, wiring diagrams, and the VLAB.
  • VLABThe whole fabric — controller + virtual SONiC switches — as VMs on one Linux box.
  • Wiring DiagramThe YAML set describing physical truth: switches, servers, connections.
  • GatewayHedgehog's edge component for NAT/PAT between VPCs and external networks.

Underneath

  • SONiCMicrosoft-born, Linux-Foundation-governed open switch OS; runs Azure. Community and Broadcom Enterprise flavors.
  • ONIEThe bootloader environment on white-box switches that network-installs a NOS — ZTP's first rung.
  • EVPN / VXLANThe standard overlay pair: BGP EVPN distributes tenant routes; VXLAN encapsulates tenant frames.
  • gNMIgRPC-based config/telemetry protocol used to program and stream from the switch.
  • Celestica / Edgecore etc.White-box switch makers (OCP designs) that SONiC targets.

Neighbors & rivals

  • NetrisCommercial NAAM platform: same tenancy goal, adds InfiniBand/NVLink/DPU orchestration and the SoftGate tenant edge.
  • Metal³ ("metal kubed")The same CRD-reconciler pattern applied to servers (BareMetalHost) instead of switches.
  • NVIDIA Spectrum-X / BlueFieldThe AI Ethernet reference stack and the DPU that extends fabric isolation inside the server.
  • UFM / NMXNVIDIA managers for InfiniBand partitions (PKeys) and NVLink partitions — the non-Ethernet tenancy planes.
  • OCPOpen Compute Project — publishes the open cluster/fabric reference architectures Hedgehog aligns to.

Gotchas — read before trusting production to it

  1. Isolation starts at the first port you own — not before. If a tenant server carries multiple tenants (VMs, containers, multi-GPU splits), the VLAN tag is applied on the host, and a root tenant can forge it. That's exactly the gap DPUs close: a BlueField-class NIC is operator-controlled hardware inside the tenant's server. Whole-server tenancy: switch enforcement is enough. Sub-server tenancy: you need the DPU story.
  2. The control node is your new blast radius. Whoever can write CRDs can re-plumb tenancy. Treat the Kubernetes API like a switch admin credential: RBAC per resource type, admission review, Git-only writes, and an out-of-band management network that the fabric itself doesn't depend on (avoiding circular dependencies is why the OOB network stays unmanaged).
  3. Learn the exact isolation semantics before promising them. isolated seals subnets from each other within a VPC; restricted seals hosts within a subnet; permit lists re-open isolated subnets but do not affect VPC peering. Mixing these up in a customer contract is an expensive mistake.
  4. Peering is routing, so addresses must not overlap. Two VPCs can happily reuse 10.0.1.0/24 — until you peer them. IPv4Namespaces exist to manage exactly this; plan shared-services addressing (storage VPC) in a range no tenant will ever be given.
  5. Hardware compatibility is a real list, not "anything SONiC". Fabric features (sub-interfaces for external peering, VXLAN scale, RoCE profiles) vary by ASIC and SONiC build. Check the supported-devices list before buying switches, and expect version coupling between Fabric releases and SONiC images.
  6. Reconciliation cuts both ways. Out-of-band CLI "quick fixes" on a switch get reverted by the agent — by design. Your on-call runbooks must go through the API, and your engineers must unlearn the SSH-to-the-switch reflex.
  7. Lab honestly before hardware. The virtual lab (next section) runs the same controller and SONiC images as production — use it to rehearse tenant onboarding, peering, and failure drills before a single cable exists.

Run it yourself

LAB A — Compile a fabric on your laptop (no VMs)

~10 min · Linux or macOS · free

  1. Install the Hedgehog CLI (hhfab) and its fetch helper ORAS.

    curl -fsSL https://i.hhdev.io/oras  | bash
curl -fsSL https://i.hhdev.io/hhfab | bash
hhfab --version

    EXPECT: a version string like hhfab version v0.3x.x

  2. Initialize a dev config and generate a wiring diagram — the declarative description of an entire spine-leaf fabric.

    hhfab init --dev
hhfab vlab gen
less vlab.generated.yaml

    EXPECT: YAML full of Switch, Server, and Connection objects — the fabric's physical truth as Kubernetes resources

  3. Read it as a tenant-isolation engineer. Find a Connection of type mclag; that name is exactly what a VPCAttachment would reference (Section 4, tab 2). You've just traced intent → wiring without booting anything.

    EXPECT: you can answer "which two leaves would enforce tenant-a's VRF for server-01?"

LAB B — Full virtual fabric: create two tenants, prove isolation

~1–2 h · Linux x86 host · ≈22 vCPU / 20 GB RAM (collapsed-core)

  1. Bring up the VLAB — real Fabric Controller, virtual SONiC switches, QEMU test servers. Use the smaller collapsed-core topology; a big cloud VM works fine.

    hhfab init --dev --fabric-mode collapsed-core
hhfab vlab gen
hhfab vlab up        # foreground; Ctrl-C tears it down

    EXPECT: after several minutes, log lines showing control node ready and switches provisioned

  2. Create two VPCs and attach one server to each (second terminal; hhfab vlab ssh → control-1; user core, password HHFab.Admin!).

    kubectl fabric vpc create --name vpc-1 --subnet 10.0.1.0/24 --vlan 1001 --dhcp
kubectl fabric vpc create --name vpc-2 --subnet 10.0.2.0/24 --vlan 1002 --dhcp
kubectl fabric vpc attach --vpc-subnet vpc-1/default --connection server-01--mclag--leaf-01--leaf-02
kubectl fabric vpc attach --vpc-subnet vpc-2/default --connection server-02--mclag--leaf-01--leaf-02

    EXPECT: kubectl get vpc shows both; get agents shows switch generations advancing as configs land

  3. Prove the isolation, then break it deliberately. From server-01 (DHCP an address first), ping server-02 — it must fail. Then create a peering and watch the same ping succeed within seconds.

    # on server-01:  ping 10.0.2.x        → 100% loss  (isolated by default)
# on control-1:
kubectl fabric vpc peer --vpc vpc-1 --vpc vpc-2
# on server-01:  ping 10.0.2.x        → replies     (explicit sharing)

    EXPECT: the FIG. 3-1 story, reproduced by you: default-deny in silicon, opened only by a named API object

LAB C — Graduation: operate it like a neocloud

open-ended · same VLAB

  1. Rehearse the gotchas. SSH to a virtual switch (user admin), hand-edit something, and watch reconciliation revert it. Then try the isolation flags: recreate vpc-1 with defaultRestricted: true and observe two servers in the same subnet unable to reach each other.

    EXPECT: visceral understanding of "the API is the only door"

  2. Do it the GitOps way. Move your VPC YAMLs into a Git repo and apply via Flux or Argo CD; tenant onboarding becomes a pull request. Then work through the free labs at hedgehog.cloud/learn — reconciliation tracing, observability, Day-2 ops.

    EXPECT: a repo where "create a tenant" is a reviewed, revertable commit — the operating model this whole sheet is about

Glossary — the jargon, decoded

VPC — Virtual Private Cloud
A tenant's isolated private network: subnets, VLANs, optional DHCP. Deliberately mirrors the AWS concept; in the fabric it compiles to a VRF plus VXLAN segments.
VRF — Virtual Routing & Forwarding
Multiple independent routing tables coexisting on one switch. Two VRFs can hold identical IP ranges and never see each other — the core mechanism of hard tenancy. wikipedia
VXLAN / VNI
VXLAN wraps tenant Ethernet frames in UDP so they cross a routed fabric; the VNI (VXLAN Network Identifier) is the 24-bit tenant/segment tag on every packet. Done in the switch ASIC at line rate. wikipedia
EVPN
BGP address family that distributes tenant MAC/IP (Type-2) and prefix (Type-5) routes fabric-wide, scoped by route-targets so each VRF imports only its own tenant's routes. wikipedia
VTEP
VXLAN Tunnel EndPoint — the device (here, each leaf switch) that encapsulates/decapsulates tenant traffic into VXLAN tunnels.
CRD — Custom Resource Definition
Kubernetes' extension mechanism: define a new object type (VPC, Switch…) and the API server stores/validates instances of it, ready for a controller to act on. k8s docs
Reconciliation
The control-loop pattern: continuously compare desired state (the CRD) with observed state (the switch) and correct the difference — the opposite of fire-and-forget config push.
SONiC
Open-source network OS (Linux + a Redis config database + switch drivers) started by Microsoft for Azure, now Linux Foundation governed. Runs on many white-box switch models. sonicfoundation.dev
ONIE
Open Network Install Environment — a small pre-installed boot environment on white-box switches that fetches and installs a network OS over the network. onie docs
ZTP — Zero-Touch Provisioning
Rack it, cable it, power it — the switch installs its OS (via ONIE) and joins the fabric with no human at a console.
gNMI
gRPC Network Management Interface — a modern protocol for setting device config and streaming telemetry, used to program SONiC switches. openconfig/gnmi
Border leaf
A leaf switch designated to exchange routes with the outside world (upstream routers/firewalls) — where ExternalPeering lands and tenant traffic exits the fabric.
DPU / SmartNIC
A NIC with its own CPUs (e.g. NVIDIA BlueField) that the operator controls even inside a tenant's server — extending fabric-grade isolation to sub-server granularity (single GPUs, VMs).
MCLAG / ESLAG
Ways to dual-home one server to two leaf switches so either switch can fail without the server losing connectivity; Connections in the wiring diagram carry these types.
Leaf-spine (Clos)
The standard DC topology: every leaf (top-of-rack) connects to every spine; servers only touch leaves; any-to-any traffic is at most leaf→spine→leaf with ECMP across spines.
Metal³ ("metal kubed")
CNCF-hosted project applying the same CRD-and-controller pattern to servers: a BareMetalHost resource per machine, provisioned by a controller (using OpenStack Ironic underneath). metal3.io
