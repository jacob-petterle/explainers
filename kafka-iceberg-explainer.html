1

What they are

The moverKafka

A conveyor belt with a memory.

Kafka is a durable, append-only log, split into partitions for scale. Producers append records; consumer groups read at their own pace, each tracking an offset — so a slow reader never blocks a fast writer, bursts get absorbed, and multiple independent consumers replay the same stream.

Modern Kafka runs in KRaft mode — the cluster manages its own metadata; ZooKeeper is gone.

The organizerIceberg

A database's brain grafted onto plain files.

Iceberg is a table format: your data stays as ordinary Parquet files in object storage, while Iceberg's metadata layer (snapshots, manifests) plus a catalog adds ACID transactions, safe schema evolution, time travel, and hidden partitioning.

Result: any SQL engine — Spark, Trino, DuckDB, Snowflake — can query the same tables. The files are yours; no engine owns them.

The relationship in one sentence: Kafka moves and buffers, Iceberg organizes and remembers, and a sink connector is the bridge that turns a stream into tables — exactly once.

The job it solves

Hot store keeps weeks; you need yearsMetrics engines are deliberately short-horizon. The lakehouse is where "compare this June to last June" lives.
Burst absorptionA fleet-wide event storm can't overwhelm downstream writers — the log buffers, consumers drain at their pace.
One stream, many readersThe same metrics feed a table sink, an anomaly detector, and an alert enricher — independently, replayably.
Correlation studies"Thermals vs job performance over six months, per hardware batch" — a SQL join over billions of rows, offline, cheap.
Vendor-neutral historyOpen files + open format: switch query engines anytime; the data never migrates.
Training data for optimizationClean, partitioned, timestamped fleet telemetry is exactly the feedstock for the models that tune it.
Why not just a big database? Cost and coupling. Object storage is ~10× cheaper than database disks, scales without resharding, and Iceberg keeps the data in an open format no single engine owns. Databases stay for the hot tier; the lakehouse is for the ocean.

Hands-on: produce, sink, query, time-travel

# The whole Kafka mental model in two commands:
kafka-console-producer --topic fleet.metrics \
  --bootstrap-server localhost:9092
> {"serial":"7X2K9M3","temp_c":46.5,"ts":"…"}

kafka-console-consumer --topic fleet.metrics \
  --from-beginning --bootstrap-server localhost:9092
# ← replays EVERYTHING still in retention:
#   the log remembers; consumers just move a cursor
Design choice that matters: the record key (e.g. serial) decides the partition — same key, same partition, guaranteed order per machine. Choose keys like you choose primary keys.
# Kafka Connect: the bridge is JSON config, not code
{
  "connector.class":
    "org.apache.iceberg.connect.IcebergSinkConnector",
  "topics": "fleet.metrics",
  "iceberg.tables": "fleet.metrics_raw",
  "iceberg.catalog.type": "rest",
  "iceberg.catalog.uri":  "https://…/catalog",
  "iceberg.control.commit.interval-ms": "300000"
}
The sink batches records into Parquet and commits an Iceberg snapshot every interval — exactly-once, with commit coordination and multi-table fan-out built in. Crash-safe because offsets live in Kafka itself.
-- Any engine, same table. The correlation study:
SELECT hw_batch,
       avg(temp_c)             AS avg_temp,
       avg(job_tokens_per_sec) AS perf
FROM   fleet.metrics_raw m
JOIN   fleet.job_stats j USING (serial, day)
WHERE  m.day BETWEEN DATE '2026-01-01' AND DATE '2026-06-30'
GROUP  BY hw_batch
ORDER  BY perf DESC;

-- partition pruning: only the needed day-files are read
Six months × whole fleet ≈ billions of rows — and it's a cheap offline query, because hidden partitioning by day(ts) means engines skip every file outside the range.
-- Every commit is a snapshot. Query the past:
SELECT count(*) FROM fleet.metrics_raw
  FOR TIMESTAMP AS OF TIMESTAMP '2026-07-01 00:00:00';

-- inspect the table's own history:
SELECT snapshot_id, committed_at, operation
FROM   fleet.metrics_raw.snapshots;

-- and schema evolution is a metadata edit, not a rewrite:
ALTER TABLE fleet.metrics_raw ADD COLUMN gpu_power_w double;
Time travel + snapshot history is what "ACID on files" buys you: reproducible analyses ("the exact data the model trained on"), safe experiments, and painless audits.

Things worth knowing before you lean on it

  • Kafka retention is a buffer, not the archive. Default topic retention (days) exists to absorb and replay recently — the durable copy is Iceberg. Size retention to survive your longest plausible sink outage, not to hoard.
  • Partitions and keys are near-permanent decisions. The key determines ordering and distribution; repartitioning later is a migration. Rule: key by the entity whose events must stay ordered (a machine's serial), and start with more partitions than brokers.
  • The small-file problem is guaranteed — schedule compaction. Streaming sinks commit frequently, producing many small Parquet files that slow every query. R2 Data Catalog compacts managed tables automatically; self-managed setups run a periodic Spark/maintenance job, plus snapshot expiry so metadata doesn't grow forever.
  • One catalog to rule them all. The catalog is where "the table" actually lives; two catalogs pointing at one bucket is how lakehouses corrupt. Pick one (REST-standard) early and route every writer and reader through it.
  • Monitor consumer lag like a smoke alarm. Lag = how far behind a consumer group is. A silently stuck sink plus finite retention equals a hole in your history — alert on lag growth, not just failures.
  • Exactly-once has fine print. The Iceberg sink coordinates commits for exactly-once into tables; your producers still need sane retries (idempotent producer settings) so duplicates don't enter the log upstream.
  • Schema is a contract — manage it. The sink can evolve tables automatically, but garbage records evolve garbage columns. A schema registry (or strict producer validation) at the front door pays for itself the first time someone renames a field.
  • Kafka is stateful infrastructure. KRaft removed ZooKeeper, not the ops: it's a replicated cluster (Strimzi operator on K8s, or managed) with disks, upgrades, and monitoring — budget for it or buy it.

Run it yourself

LAB A — Kafka on your laptop

~15 min · needs Docker

  1. Run a single-node KRaft broker (the official image needs no ZooKeeper, no config):

    docker run -d --name kafka -p 9092:9092 apache/kafka

  2. Produce and consume. Predict first: if you start the consumer with --from-beginning after producing, what appears?

    docker exec -it kafka /opt/kafka/bin/kafka-console-producer.sh \
  --topic fleet.metrics --bootstrap-server localhost:9092
# type a few JSON lines, Ctrl-C, then:
docker exec -it kafka /opt/kafka/bin/kafka-console-consumer.sh \
  --topic fleet.metrics --from-beginning \
  --bootstrap-server localhost:9092

    EXPECT: everything you typed replays — the log remembered; the consumer just moved a cursor. That's the whole durability model.

LAB B — A lakehouse in one Python file

~25 min · pip only, no cluster

  1. Create a local Iceberg table with PyIceberg (SQLite catalog, files on disk — the real format, zero infrastructure):

    pip install "pyiceberg[sql-sqlite,pyarrow]" duckdb
python - <<'EOF'
from pyiceberg.catalog.sql import SqlCatalog
import pyarrow as pa
cat = SqlCatalog("lab", uri="sqlite:///cat.db",
                 warehouse="file://./warehouse")
cat.create_namespace("fleet")
t = cat.create_table("fleet.metrics", schema=pa.schema(
    [("serial", pa.string()), ("temp_c", pa.float64())]))
t.append(pa.table({"serial":["7X2K9M3","8Y3L0N4"],
                   "temp_c":[46.5, 41.2]}))
EOF

    EXPECT: a ./warehouse folder containing Parquet data files AND a metadata/ tree — Iceberg's anatomy, inspectable with ls.

  2. Query the same table from a different engine — the open-format promise, demonstrated:

    python - <<'EOF'
from pyiceberg.catalog.sql import SqlCatalog
import duckdb
t = SqlCatalog("lab", uri="sqlite:///cat.db",
               warehouse="file://./warehouse").load_table("fleet.metrics")
print(duckdb.sql("SELECT * FROM t.scan().to_arrow() tbl"))
EOF

    EXPECT: your rows back via DuckDB. Append more rows and re-query; then peek at t.history() — snapshots accumulating.

LAB C — The real bucket (managed catalog)

~30 min · a Cloudflare account (or any S3+REST catalog)

  1. Enable a managed Iceberg catalog on an R2 bucket and point PyIceberg at the catalog URI it prints:

    npx wrangler r2 bucket create fleet-lake
npx wrangler r2 bucket catalog enable fleet-lake

    Auth and connection details in the R2 Data Catalog docs. Repeat Lab B's write/read against it — same code, real object storage, compaction managed for you.

  2. Production shape from here: Kafka via the Strimzi operator on your management cluster, the Iceberg sink connector on Kafka Connect pointing at that catalog, a lag alert on the sink's consumer group, and a weekly snapshot-expiry job. That's the whole memory of the fleet, running.

Glossary & further reading

Topic / log
Kafka's unit of organization: a named, durable, append-only sequence of records. Writers append; readers never remove.
Partition
A topic's shards — each an ordered log. Order is guaranteed within a partition only; the record key decides which partition.
Offset
A consumer's bookmark: the position in a partition it has read to. Rewind it and the past replays.
Producer / consumer group
Writers, and named teams of readers that split partitions among themselves — each group gets the full stream independently.
KRaft
Kafka's built-in Raft-based metadata quorum — the modern mode; ZooKeeper is fully retired as of Kafka 4.x.
Kafka Connect
The config-driven integration runtime: connectors (like the Iceberg sink) run as fault-tolerant tasks, offsets tracked in Kafka itself.
Consumer lag
How far a consumer group trails the log's head — the health metric for any pipeline riding on Kafka.
Table format
A spec that makes files in object storage behave like a database table: schema, transactions, and file tracking. Iceberg is the neutral-ground standard.
Parquet
The columnar file format underneath: compressed, typed, readable by every analytics engine.
Snapshot / manifest
Each commit produces a snapshot — a metadata tree listing exactly which data files constitute the table at that moment. The basis of ACID and time travel.
Catalog (REST)
The index that maps table names to current metadata — the single place writers commit through and readers resolve through. R2 Data Catalog is a managed one built into a bucket. docs
Time travel
Querying a table as of a past snapshot or timestamp — reproducibility for audits and ML training runs.
Compaction
Maintenance that rewrites many small files into few large ones (and expires old snapshots) — the tax streaming ingestion pays for query speed.
Hidden partitioning
Iceberg partitions by transforms like day(ts) without polluting your schema — queries prune files automatically, and the partitioning can evolve later.
