What it is

The Fabric — hardware sideU-NET1

A cloud network built from switches you own.

Physically, a Hedgehog fabric is a spine-leaf network of commodity switches from Dell, Celestica, Edgecore, or Supermicro — all Broadcom silicon — running Broadcom Enterprise SONiC. Servers plug into leaves; leaves plug into spines.

Between the switches: an eBGP underlay (the hyperscaler routing design from RFC 7938). On top: a VXLAN/EVPN overlay that makes isolated virtual networks span any set of switches.

The control plane — software sideCRD-K3S

Kubernetes, but the pods are switches.

One control node runs K3s. Every physical thing (Switch, Server, Connection — together the wiring diagram) and every logical thing (VPC, peering, external) is a Kubernetes CRD.

A Fabric Controller compiles your intent into per-switch desired state; a Fabric Agent on every switch applies it via gNMI and reconciles forever. Switches ship logs/metrics home. Install, discovery, and upgrades are handled by hhfab (the Fabricator).

The relationship in one sentence: you declare the network you want as YAML; the controller compiles it into switch state; agents on each switch make reality match, continuously. No SSH-ing into switches, no per-box CLI.

2

The job it solves

Hedgehog exists for teams who need cloud-provider-style networking on their own metal — especially GPU/neocloud operators renting capacity to multiple customers — without a networking department:

You rent GPUs to strangersEach customer gets a VPC: their own subnets, DHCP, and hardware-enforced isolation on shared switches.
Nobody on staff speaks BGPUnderlay ASNs, VNIs, VRFs, route-targets — all derived automatically from a few YAML objects.
New rack of switches arrivesPlug in, power on: switches netboot via ONIE from the control node, install SONiC, self-configure. Zero touch.
GPU traffic needs lossless Ethernetroce: true on a switch reconfigures buffers and QoS for RDMA — no hand-tuned QoS maps.
Tenant needs internet / DC egressBorder leaves peer BGP with your edge; Gateway nodes add NAT and firewalling per VPC.
Vendor lock-in allergyOpen source (Apache-2.0), commodity Broadcom switches from four vendors, OCP-published reference designs.
The core promise: a small platform team declares VPCs and attachments the way they'd declare Kubernetes Deployments — and gets an AWS-like multi-tenant network, on owned hardware, that keeps itself configured.

What it actually does

  • ZTP / BOOTSwitches are discovered, imaged (ONIE), and enrolled automatically by the control node's boot + DHCP services.
  • VPCPer-tenant virtual networks: named subnets, VLANs, per-subnet DHCP (with PXE options), static routes, isolation flags.
  • ATTACHVPCAttachment binds a VPC subnet to a server-facing connection — that switch port now lives in the tenant's network.
  • MULTI-HOMEServers can dual/quad-home to 2–4 leaves via ESLAG (EVPN multi-homing) for switch-failure tolerance.
  • PEERVPCPeering selectively connects tenant networks; ExternalPeering connects a VPC to the outside via BGP on border leaves.
  • GATEWAYDedicated gateway nodes (Rust/DPDK dataplane + FRR) add NAT, port-forwarding, and stateful edge services per VPC. GA since 26.01.
  • ROCELossless-Ethernet profile for RoCEv2 GPU traffic: DSCP-based queueing, ECN, plus QPN-aware ECMP on Tomahawk 4G/5 silicon.
  • OBSERVEEvery switch runs Grafana Alloy; logs and metrics stream to any Prometheus/Loki-compatible stack you point it at.
  • UPGRADEOne-command fabric upgrades (from the immediately preceding release), including switch agents and control plane.
  • VLABThe entire fabric — ONIE boot and all — runs virtually under QEMU/KVM for free experimentation. See the labs in §12.

Terminology decoder — node vs. cluster vs. machine vs. tenant

Hedgehog's vocabulary is small and deliberately network-centric. Most confusion comes from words that mean something in your world (Kubernetes, GPU clusters) but map differently — or not at all — in Hedgehog's object model:

You say…Hedgehog objectWhat it actually means here
machine / host / GPU boxServerAny physical box cabled into the fabric — GPU nodes, storage, even the control node itself. Its spec is nearly empty (a description and an unused profile): the fabric only cares about its cables, which are Connection objects.
nodeControlNode / FabNodeAmbiguous — three meanings. ControlNode: the K3s management host. FabNode: a Hedgehog-managed host with a role (today the only role is gateway). And in casual K8s-speak, "nodes" are whatever joined the K3s cluster (control + gateways). Your GPU nodes are never "nodes" to Hedgehog — they're Servers.
cluster(no object)Hedgehog has no cluster concept. Your compute cluster (Kubernetes, Slurm, Ray) is invisible to it — it sees only NICs on switch ports. When Hedgehog docs say "cluster," they mean the little K3s management cluster. The network as a whole is called the Fabric.
tenantVPCThe only tenancy primitive. There is no Tenant, Project, or Org CRD, and no per-tenant API accounts — a "tenant" is just a VPC (or a few) that you mentally assign to a customer. See §7 for what that does and doesn't buy you.
switchSwitchA SONiC device with a role: spine, server-leaf, border-leaf, or mixed-leaf — plus a SwitchProfile that describes what the hardware can do (§6).
cabling / topologyWiring DiagramNot one CRD but the whole set of Switch + Server + Connection (+ namespace) objects, usually kept in one YAML file, validated by hhfab before install. It is the machine-readable as-built of your network.
the networkFabricBoth the product's name for the whole system and a config value: fabricMode has exactly one value today, spine-leaf (mesh is a variant of it, see §6). One control plane manages exactly one fabric.
The mental model: Hedgehog manages switches, knows servers only as cable endpoints, expresses tenants only as VPCs, and has no idea your GPUs form a cluster. Anything inside the server — VMs, containers, which tenant's job is running — is out of scope and someone else's problem (yours).

Hands-on — what the YAML actually looks like

Six objects cover 90% of daily life. Tabs below are trimmed from real examples in the official docs and repo (githedgehog/docs, githedgehog/fabric):

# A switch + a dual-homed server, from the wiring diagram
apiVersion: wiring.githedgehog.com/v1beta1
kind: Switch
metadata: { name: leaf-01 }
spec:
  role: server-leaf          # spine|server-leaf|border-leaf|mixed-leaf
  profile: dell-s5248f-on    # must match a SwitchProfile
  redundancy: { group: eslag-1, type: eslag }
  portBreakouts: { "E1/55": "4x25G" }
---
kind: Connection
metadata: { name: server-2--eslag--leaf-01--leaf-02 }
spec:
  eslag:                     # LAG across 2–4 leaves (EVPN MH)
    links:
      - server: { port: server-2/enp2s1 }
        switch: { port: leaf-01/E1/1 }
      - server: { port: server-2/enp2s2 }
        switch: { port: leaf-02/E1/1 }
The wiring diagram is your as-built, as code. hhfab validate catches miscabling before install, and the admission webhook keeps rejecting invalid changes forever after.
# A tenant network: two subnets, DHCP, hard isolation inside
apiVersion: vpc.githedgehog.com/v1beta1
kind: VPC
metadata: { name: tenant-a }
spec:
  ipv4Namespace: default   # guarantees no overlapping CIDRs
  vlanNamespace: default   # guarantees no VLAN collisions
  mode: ""                  # "" = L2VNI (default) · l3vni · l3flat
  subnets:
    default:
      subnet: 10.10.1.0/24
      vlan: 1001
      dhcp: { enable: true, range: { start: 10.10.1.10 } }
    gpu-backend:
      subnet: 10.10.2.0/24
      vlan: 1002
      isolated: true    # cut off from other subnets in this VPC
      restricted: true  # hosts inside can't even see each other
      hostBGP: true     # hosts may BGP-peer with their leaf
One VPC = one VRF on every switch that needs it, one VNI per subnet — all allocated automatically. isolated/restricted/permit are the intra-tenant firewall knobs (§7).
# The moment a physical server joins a tenant network
apiVersion: vpc.githedgehog.com/v1beta1
kind: VPCAttachment
metadata: { name: tenant-a-server-2 }
spec:
  subnet: tenant-a/default   # which VPC + subnet
  connection: server-2--eslag--leaf-01--leaf-02
  nativeVLAN: false          # true = untagged on the port

# or, one-liner from the control node:
kubectl fabric vpc attach --vpc-subnet tenant-a/default \
    --connection server-2--eslag--leaf-01--leaf-02
This is the isolation enforcement point. Reassigning a server from tenant-a to tenant-b is: delete one VPCAttachment, create another. The switch port flips VRFs; the server needs only a DHCP renew.
# Let two tenants exchange traffic — selectively
kind: VPCPeering
metadata: { name: tenant-a--tenant-b }
spec:
  permit:
    - tenant-a: { subnets: [default] }
      tenant-b: { subnets: [default] }
---
# …or reach the outside world via a border leaf
kind: ExternalPeering
spec:
  permit:
    vpc: { name: tenant-a, subnets: [default] }
    external: { name: upstream-isp,
                prefixes: [{ prefix: 0.0.0.0/0 }] }
Default-deny everywhere: no VPC reaches anything — not even another VPC on the same leaf — until a peering explicitly permits specific subnets/prefixes. Route-leaking happens in hardware, at full fabric bandwidth.
# Lossless Ethernet for RDMA/GPU traffic: one flag
kind: Switch
metadata: { name: rail-leaf-01 }
spec:
  profile: celestica-ds5000   # TH5 · 64×800G OSFP
  roce: true                   # lossless queues + DSCP QoS (reboot!)
  ecmp: { roceQPN: true }      # hash on RDMA queue-pair for entropy

# What roce:true configures on the switch:
#   DSCP 24/26 → traffic class 3 (lossless, RDMA)
#   DSCP 48    → traffic class 6 (CNP / congestion signals)
# On the GPU hosts, tag NCCL traffic accordingly:
export NCCL_IB_TC=96
RoCE is a per-switch boolean — Hedgehog picks the buffer/PFC/ECN profile for you. Simple, but you can't hand-tune thresholds through the API, enabling it requires a reboot, and it locks port breakouts (§11).
# PREVIEW: generate a rail-optimized GPU fabric wiring diagram
hhfab vlab generate-gpu-rail \
    --scalable-units 2 \             # SUs to stamp out
    --vpcs 2 \                       # tenant VPCs to pre-create
    --servers-per-vpc-per-scalable-unit 2

# What it emits: 2 spines · 4 leaves per SU · 8 rails;
# every GPU server gets one link per rail to a rail-leaf
# (leaf-suNN-rN), /31 point-to-points, a /16 per rail
# inside each VPC's /13, hostBGP enabled.
Rail-optimized means GPU n of every server shares a leaf with GPU n of every other server — NCCL's heaviest collective traffic stays one hop away and off the spines. This generator is the seed of Hedgehog's back-end-network story (§8) — today it's a PREVIEW tool that emits VLAB wiring.

Topologies it supports — and the hardware menu

Topologies

  • SPINE-LEAFThe primary (and only) fabricMode. Standard 2-tier Clos: every leaf connects to every spine; capacity scales by adding leaves (ports) or spines (bandwidth). Rack-scale to multi-row.
  • MESHFor 1–3 leaves: skip the spines, cable leaves directly to each other (added in 25.05). Same API and VPCs — a budget entry point for labs, edge sites, or a first rack.
  • ESLAGServer multi-homing to 2–4 leaves via EVPN ESI multi-homing — survive a leaf failure without proprietary MLAG pairs.
  • BREAKOUTSFirst-class: split an 800G port into 2×400G, a 100G into 4×25G, etc., declared per-port on the Switch object.
  • RAIL (PREVIEW)Rail-optimized GPU topology via generate-gpu-rail — 8 rails, 4 leaves per scalable unit (§5, §8).
  • GONECollapsed-core was removed (25.05) and MCLAG is deprecated (removal targeted 26.03 — use ESLAG). Docs/blogs mentioning them are historical.

The isolation & multi-tenancy model, precisely

Everything is default-deny and enforced in switch hardware (VRFs, VLANs, ACLs). Isolation stacks in four layers, coarse to fine:

  • L1 · VPCA VPC is a hermetic routing universe (VRF + VNIs). Tenant A's packets cannot reach tenant B — or the internet — no matter what hosts try. Crossing requires an explicit VPCPeering, ExternalPeering, or Gateway policy.
  • L2 · SUBNETWithin a VPC, isolated: true walls a subnet off from sibling subnets; permit lists selectively re-open pairs/groups. Think "security zones inside one tenant."
  • L3 · HOSTrestricted: true blocks host-to-host traffic within the same subnet. Yes: two servers on the same leaf, in the same subnet, can be isolated from each other — enforced at the port, not by distance.
  • L4 · EDGEBorder leaves filter external routes with BGP communities; Gateway nodes add stateful NAT (masquerade, port-forward, static) and firewalling per VPC.

Where the boundary actually sits — "machine isolation"

The enforcement point is the VPCAttachment: a binding between a VPC subnet and a server's switch port(s). So isolation is per-NIC/per-port, which for bare-metal single-tenant boxes is exactly "machine isolation" — each machine sits in its tenant's network and nothing else.

What it is not: Hedgehog runs nothing on the server, so it cannot see or separate anything inside the machine. If one physical host carries two tenants' VMs on one NIC, Hedgehog can't tell them apart (beyond trunking different VLAN subinterfaces to different subnets — still your hypervisor's job to keep honest). GPU-level isolation, MIG, VM security: out of scope.

The RBAC gap, stated plainly: tenancy is a data-plane concept here. The Kubernetes API that defines VPCs is one trusted admin surface — there are no per-tenant accounts, quotas, or namespaced permissions in the CRDs. If customers need self-service ("create my own subnets"), you build that portal/API yourself and let it talk to Hedgehog. This is the single biggest gap for anyone building a GPU cloud on it (§11).

The B200-cluster lens — could you bring one up on this?

A GPU cluster has (at least) two networks. Judge Hedgehog separately on each:

Front-end / converged networkREADY

This is Hedgehog's home turf.

Provisioning, storage, tenant traffic, internet: Trident-3 leaves (25/100G) or DCS240 (400G), ESLAG dual-homing, one VPC per tenant, DHCP+PXE per subnet, border-leaf BGP + Gateway NAT for egress. Production deployments (neoclouds like FarmGPU) run this today, and the design is published as OCP reference architectures (OPG-64 → OPG-1024 xPUs).

Back-end / GPU RDMA networkEMERGING

The pieces exist; the polish is recent.

An HGX B200 node exposes 8×400G NICs — one per GPU, one rail each. The Hedgehog recipe: DS5000 (TH5, 64×800G) rail-leaves with ports broken out to 2×400G, roce: true (lossless DSCP/ECN), ecmp.roceQPN: true for flow entropy, L3VNI-mode VPCs with hostBGP, and the generate-gpu-rail topology (§5). Hedgehog's own blog argues this L3-to-the-host design beats stretched L2 for RoCE fabrics.

Back-of-envelope: one scalable unit for B200s

  • SERVERS32 HGX B200 nodes ⇒ 256 GPUs, 256×400G rail ports (8 per node).
  • RAIL LEAVES8 rails → with 2×400G breakout a DS5000 serves 128×400G, so rails pair up: 4 DS5000 leaves per SU (the generator's default: 4 leaves, 8 rails, 2 rails per leaf).
  • SPINES2+ DS5000/DS4101 spines; add spines for more bisection, add SUs to scale out — OCP blueprints go to 1,024 xPUs.
  • TENANCYRail network per tenant = a VPC with a /16 per rail; isolation and re-assignment identical to §7 — this is the differentiator vs. hand-built RoCE fabrics.
  • HOSTSYour job: NCCL env (NCCL_IB_TC=96), /32-style addressing quirks of L3VNI mode (gratuitous-ARP/arp_notify on hosts), optional FRR for hostBGP.
Honest verdict: front-end network for a B200 cluster — yes, today, comfortably. Back-end RDMA fabric — technically supported (RoCE + 800G TH5 + QPN hashing are real, shipped features) and OCP-blessed on paper, but the rail topology tooling is marked PREVIEW, RoCE tuning is take-it-or-leave-it, and you should expect to be an early adopter validating NCCL performance yourself rather than inheriting a decade of Spectrum-X/InfiniBand tuning. Plan a proof-of-concept, not a copy-paste.

Gotchas & gaps — read before committing

  • No per-tenant RBAC or self-service. The whole API is one trusted-admin surface (§7). A GPU cloud needs its own portal in front, mapping customers → VPCs and gating every change. Budget for building it.
  • Single control node; HA is "(Future)". Dataplane survives a control-node outage, but config changes, DHCP for new leases, and switch boot don't. Back up the node; plan its recovery story.
  • IPv4 only. No IPv6 in VPCs, underlay, or externals — validators literally reject non-IPv4 CIDRs. Disqualifying for some clouds; check your requirements now, not later.
  • The 800G box bends the model. DS5000 (TH5) as a leaf is L3-only: VPCs in L3VNI mode, no stretched-L2, no ESLAG, hosts need arp_notify/GARP habits. Your shiny GPU fabric uses the newest, least-worn code paths.
  • RoCE is coarse and disruptive. One boolean, fixed DSCP/queue profile, switch reboot to enable, breakouts locked while on. Fine for standard NCCL; a wall if you need custom PFC/ECN tuning.
  • Deprecations in flight. MCLAG and remote VPC peering die in 26.03 — avoid in new designs. Collapsed-core is already gone. Old blog posts and LLM training data will mislead you here.
  • Upgrades are single-step. 26.02 upgrades only from 26.01.x — fall two releases behind and you're doing multiple hops. SONiC itself isn't upgraded by fabric upgrades. Known SONiC-level bugs (stale-VNI on VPC recreate, DS5000 transceiver init) are documented in known limitations.
  • One fabric per control plane, one deployment per site. No multi-fabric federation object; inter-site is Gateway/External territory. Scale guidance is bucketed at <50 / >50 switches — beyond that you're pioneering.
  • The seductive demo is virtual. VLAB proves the API and workflow, not the dataplane — SONiC-VS forwards in software. Performance claims need metal. Which is exactly what the labs below are for: burn the workflow into your hands for free, then decide what to buy.

Run it yourself — zero switches required

Hedgehog's virtual lab is unusually honest: switches boot from ONIE images and install SONiC-VS exactly like hardware would. Ladder: read the API on any Linux box → run a whole fabric in VMs → graduate to metal.

LAB A — Generate & read a fabric (no VMs)

~15 min · any Linux/WSL box · needs Docker

  1. Get credentials, then install hhfab — Hedgehog's all-in-one CLI (source: githedgehog/fabricator). Downloads need a free token from the download request form:

    docker login ghcr.io --username <provided> --password <token>
curl -fsSL https://i.hhdev.io/hhfab | bash

    EXPECT: hhfab installed; `hhfab --version` prints a 26.x version.

  2. Initialize a dev config and generate the default topology (2 spines, 2 ESLAG leaves + 1 orphan leaf):

    hhfab init --dev
hhfab vlab gen

    EXPECT: "Building VLAB wiring diagram fabricMode=spine-leaf" → "Generated wiring file name=vlab.generated.yaml".

  3. Read the wiring diagram you just made. Everything from §4 and §5, generated for you:

    grep 'kind:' vlab.generated.yaml | sort | uniq -c
less vlab.generated.yaml   # Switch, Server, Connection/eslag…

    EXPECT: a census of Switches, Servers, Connections — the CRD tree of FIG. 4-1, in the flesh.

  4. Now the GPU version. Generate the PREVIEW rail-optimized fabric from §8 and diff the design:

    hhfab init --dev -f
hhfab vlab generate-gpu-rail --scalable-units 1 --vpcs 2
grep -c 'kind: Connection' vlab.generated.yaml   # count the rails

    EXPECT: 4 rail-leaves, 8 rails, servers with one Connection per rail, pre-built VPCs with a /16 per rail.

LAB B — Boot the whole fabric in VMs

~2–3 h · Linux + KVM · ≈54 vCPU / 50 GB RAM / 550 GB NVMe (Ubuntu 24.04 tested)

  1. Prereqs, then bring it up. Install qemu-kvm socat and Docker, reuse Lab A's directory, and:

    hhfab vlab up    # foreground; Ctrl+C tears it all down

    EXPECT: VMs boot — control node installs K3s, virtual switches ONIE-boot, install SONiC-VS, agents enroll. Wait for all switches Ready (tens of minutes).

  2. SSH to the control node and meet the API (in a second terminal):

    hhfab vlab ssh    # pick control-1
kubectl get switch,server,connection
kubectl get agents   # per-switch state: NOS version, heartbeats

    EXPECT: your wiring diagram, live, with agent heartbeats ticking.

  3. Create two tenants and attach servers — the §5 YAML via the convenience CLI:

    kubectl fabric vpc create --name vpc-01 --subnet 10.0.1.0/24 \
    --vlan 1001 --dhcp --dhcp-start 10.0.1.10
kubectl fabric vpc create --name vpc-02 --subnet 10.0.2.0/24 \
    --vlan 1002 --dhcp --dhcp-start 10.0.2.10
kubectl fabric vpc attach --vpc-subnet vpc-01/default \
    --connection server-01--eslag--leaf-01--leaf-02
kubectl fabric vpc attach --vpc-subnet vpc-02/default \
    --connection server-02--eslag--leaf-01--leaf-02

    EXPECT: agents reconfigure the virtual switches; servers pull DHCP leases from their VPC subnets.

  4. Prove the isolation model. From server-01, ping server-02 (different VPC) — it must fail. Then:

    kubectl fabric vpc peer --vpc vpc-01 --vpc vpc-02

    EXPECT: ping fails → peering applied → ping succeeds. That's §7, L1, demonstrated end-to-end by you.

Glossary — the jargon, decoded

SONiC
"Software for Open Networking in the Cloud" — the open-source Linux switch OS started at Microsoft. Hedgehog runs the commercial Broadcom Enterprise SONiC build on all supported switches. sonicfoundation.dev
Spine-leaf / Clos
Two-tier network: every leaf (top-of-rack) connects to every spine. Any server is exactly two hops from any other; capacity scales sideways. wikipedia
eBGP underlay
Each switch is its own BGP autonomous system; routes between switch loopbacks propagate via ordinary BGP — the hyperscaler design from RFC 7938. Hedgehog assigns all ASNs automatically. RFC 7938
VXLAN
Encapsulation that wraps tenant Ethernet frames in UDP so they can cross the routed underlay — the tunnels that let a VPC span switches. RFC 7348
EVPN
The BGP-based control plane that tells every switch which MAC/IP lives behind which VXLAN tunnel — no flood-and-learn. Also provides ESI multi-homing (ESLAG). RFC 8365
VNI / VTEP / VRF
VNI: the 24-bit ID naming a virtual network inside VXLAN packets (one per VPC + one per subnet). VTEP: the switch endpoint that en/decapsulates tunnels (Hedgehog leaves). VRF: a per-tenant routing table on a switch — one per VPC.
ESLAG
Hedgehog's name for EVPN ESI multi-homing: a server LAGs to 2–4 leaves that act as one logical switch, standards-based. Replaces the deprecated MCLAG (proprietary-style switch pairing).
VPC
Virtual Private Cloud — Hedgehog's tenancy unit: named subnets with VLANs, DHCP, and isolation flags, realized as a VRF+VNIs across the fabric. Deliberately shaped like the AWS concept.
L3VNI mode
VPC mode that routes everything (no stretched L2 between hosts). Required on Tomahawk 5 leaves like the DS5000, which don't do L2VNI. Hosts keep their IP but lose the illusion of a shared segment — hence GARP/arp_notify host tweaks.
hostBGP
Per-subnet option letting servers BGP-peer directly with their leaf — ECMP multihoming and VIP advertisement from the host, cloud-style L3-to-the-host.
CRD
Custom Resource Definition — a user-defined Kubernetes API type. Hedgehog's whole interface is CRDs: you manage a network with kubectl. k8s docs
K3s
A lightweight single-binary Kubernetes distribution — the management cluster on Hedgehog's control node. k3s.io
gNMI
gRPC Network Management Interface — the streaming config/telemetry protocol the Fabric Agent uses to program SONiC's config database on each switch. openconfig/gnmi
ONIE / ZTP
Open Network Install Environment — the bootloader on whitebox switches that netboots a NOS. Hedgehog's control node serves it, giving zero-touch provisioning (ZTP): rack, cable, power, done. ONIE docs
Wiring diagram
The YAML description of your physical build: every Switch, Server, and Connection. Input to hhfab; the fabric's source of truth thereafter.
Border leaf
A leaf whose job is talking to the outside: it holds ExternalAttachments and BGP-peers with edge routers Hedgehog doesn't manage.
RoCEv2
RDMA over Converged Ethernet — GPU-to-GPU memory transfers over routed Ethernet, bypassing host CPUs. Needs a "lossless" network to perform, hence PFC/ECN. What NCCL uses on Ethernet AI fabrics. wikipedia
PFC / ECN / DSCP
The lossless-Ethernet toolkit: DSCP marks packets into classes, ECN signals congestion early, PFC pauses a single priority class instead of dropping. roce: true configures a fixed profile of these (DSCP 24/26 → lossless queue, DSCP 48 → congestion notifications).
ECMP QPN hashing
Load-balancing RDMA flows across equal-cost paths by hashing the queue-pair number — more entropy for elephant RDMA flows. TH4G/TH5 silicon only (ecmp.roceQPN).
Rail-optimized
GPU-fabric topology where NIC n of every server connects to the same leaf ("rail n"). NCCL's rail-aligned collectives then stay one hop away, keeping spines uncongested. 8 rails for 8-GPU HGX nodes.
HGX B200
NVIDIA's 8×Blackwell-GPU server building block; typical configs pair each GPU with a 400G ConnectX-7/BlueField NIC — hence 8 rails × 400G per node of back-end bandwidth.
Spectrum-X
NVIDIA's Ethernet AI-networking platform (Spectrum-4 switches + BlueField DPUs, adaptive routing, telemetry congestion control). The main commercial rival for Ethernet GPU back-ends. nvidia.com
DPDK
Data Plane Development Kit — userspace packet processing that bypasses the kernel for speed. Hedgehog's Gateway dataplane is Rust on DPDK. dpdk.org
