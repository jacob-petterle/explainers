The short answer: three different jobs

"A unified view over all your hardware" is actually three separate jobs that people often blur together. Ansible and Ironic each do one of them; neither is primarily a dashboard.

Job A — Act on many machines
Ansible

Unified control. "Do this to all 400 boxes": reboot, set a BIOS flag, push firmware. Fire-and-forget tasks; no live picture, no database of state.

Job B — Manage machine lifecycles
Ironic

Unified registry + workflow. Keeps a database of every node and walks each one through enroll → deploy → recycle. An inventory, but built for provisioning — not for watching.

Job C — See everything, live
Observability & fleet consoles

The actual unified view. Prometheus Redfish exporters + Grafana, or vendor consoles (Dell OpenManage Enterprise, HPE OneView), plus a source-of-truth like NetBox.

Why any of this works at all: Redfish. Because every vendor's BMC answers the same HTTPS/JSON API, one playbook, one provisioning service, or one scraper can address a mixed fleet — Dell next to HPE next to Supermicro — without per-vendor code for the common cases.

Ansible and Ironic, up close

AutomationRed Hat / open source

A for-loop with superpowers.

Ansible runs playbooks (YAML task lists) against an inventory (your list of BMC addresses). It's agentless — for hardware work it just makes Redfish calls from a control node. Same tool many teams already use for OS config, extended down to the metal.

  • redfish_infoGather facts: model, firmware versions, health, sensors — across the fleet.
  • redfish_commandDo things: power cycle, inject virtual media, clear logs.
  • redfish_configSet things: BIOS attributes, boot order, BMC network/users.
# playbook: reboot every node in group "gpu"
- hosts: gpu_bmcs
  tasks:
    - community.general.redfish_command:
        category: Systems
        command: PowerReboot
        baseuri: "{{ inventory_hostname }}"

What it doesn't do: keep state. When the playbook ends, Ansible remembers nothing. It's a verb, not a database.

ProvisioningOpenStack / open source

Bare metal, treated like cloud.

Ironic is a long-running service with a database of nodes. Each node record holds the BMC address, credentials, driver (redfish is the standard one), and hardware profile. Ironic then drives each machine through a lifecycle — the "cattle, not pets" model applied to physical servers.

The node lifecycle it enforces:

S1ENROLLregister node + BMC creds
S2INSPECTboot agent, auto-discover CPUs, RAM, disks, NICs
S3CLEANwipe disks, reset BIOS/RAID to policy
S4AVAILABLEin the pool, ready to claim
S5ACTIVEimage written, OS deployed, handed to user

…and back to CLEAN when released. Metal³ ("metal cubed") wraps Ironic in Kubernetes resources, so clusters can claim physical machines the way they claim pods.

What it doesn't do: live dashboards or rich telemetry. Its inventory is a means to provisioning, not a monitoring system.

The realistic architecture: mature fleets run all three together. NetBox (or Ironic) as the inventory of record → feeds Ansible's inventory for actions → while a Prometheus Redfish exporter scrapes every BMC into Grafana for the live wall of dashboards. The "unified view" is that Grafana wall plus the inventory — with Ansible and Ironic as the hands that act on what you see.

Quick decision guide

  • PICK ANSIBLEYou have a list of machines and recurring tasks: firmware waves, BIOS standardization, scheduled reboots, audits. Lowest barrier to entry — a laptop and a YAML file.
  • PICK IRONICMachines come and go and you want cloud-style self-service on metal: users request a server, get a freshly imaged one, return it, it gets wiped. Heavier to run; shines at scale or under Kubernetes (Metal³).
  • PICK MONITORINGYour actual question is "is everything healthy right now?" Start with a Prometheus Redfish exporter + Grafana; add NetBox when tracking what/where becomes painful.
  • ALL THREEFull-scale fleet operations. They don't compete — they occupy the three lanes in FIG. 2-1, all speaking Redfish to the same BMCs.
