What it is

The ideadurable execution

Code with a save file.

A Temporal workflow is a normal function — loops, ifs, try/catch — but every step it takes is recorded as an event in a durable event history. If the process crashes at step 7 of 12, another worker replays the history, rebuilds the exact state, and continues from step 8 — even days later, on a different machine.

The mental flip: instead of persisting state and writing code, the code's execution is the persisted state.

The shapeserver + your workers

A brain that never runs your code.

Two halves: the Temporal Server (a cluster of Go services backed by Postgres) stores histories, runs timers, and dispatches tasks onto task queues. Workers — plain processes you write with an SDK and deploy yourself — poll those queues and execute your workflow and activity code.

Your code, credentials, and network access never leave your infrastructure; the server only ever sees inputs, outputs, and events.

The relationship in one sentence: the server remembers, your workers compute, and the event history is the contract between them.

2

The job it solves

Any process with multiple steps, real time gaps, and things that can fail — provisioning, payments, onboarding — traditionally becomes a hand-rolled state machine: a status column, a cron sweeper, retry queues, and a graveyard of stuck rows. Temporal replaces that entire genre:

The 40-minute saga"Deploy 8 machines, verify, tag, notify" — with step 6 flaking. Retries, timeouts, and resume-on-crash are built in.
The process died mid-stepDeploy your app at 2 PM; every in-flight workflow resumes exactly where it was. No orphaned state.
"Wait 30 days, then…"Durable timers survive restarts. Sleeping a month costs nothing and loses nothing.
Human in the loopWorkflow pauses for an approval signal — an hour or a week — then continues.
Undo on failureNode 6 of 8 fails permanently → compensation logic releases the other 7. The saga pattern, as a try/catch.
"What's the status?"Every execution is inspectable live — via queries, the Web UI, and full step-by-step history.
The blunt framing: without Temporal you will build a worse Temporal — status columns + cron + queues + retry bookkeeping is exactly what it is, minus the guarantees, the replay debugger, and ten years of hardening. Queues move messages; they don't remember where a four-hour process was when it died.

The vocabulary (five things cover 90%)

  • WORKFLOWThe orchestration function — deterministic code describing the steps. Can run for years; can spawn child workflows.
  • ACTIVITYA single real-world action (call NetBox, hit an API, send email) — where all side effects live. Retried automatically per policy; must be idempotent.
  • SIGNAL / QUERYAsync message into a running workflow (approve!, cancel!) / synchronous read out of it (percent complete?). Updates combine both: a tracked, validated write.
  • TIMER / SCHEDULEDurable sleep inside a workflow; Schedules are cron-with-a-brain for starting them (overlap policies, backfills, pause).
  • TASK QUEUEThe named channel matching work to worker fleets — also the unit of routing, rate control, and canary deployment for workers.

Hands-on: a provisioning saga in code

// TypeScript SDK — the orchestration, as plain code
export async function provisionCluster(order: Order) {
  const nodes = await pickNodes(order.size);      // activity
  await Promise.all(nodes.map(n =>
    deployNode(n, order.image)));                 // 10–30 min each,
                                                  // retried per policy
  await verifyCluster(order.id);                  // activity
  await tagInventory(nodes, order.tenant);        // activity
  await markReady(order.id);                      // activity → biz DB
  return { nodes };
}
This function might take 40 minutes and survive three deploys — and reads like it takes 40 milliseconds. Every await is a durable checkpoint in the event history.
// Activities: where reality gets touched. Idempotent!
export async function tagInventory(nodes, tenant) {
  for (const n of nodes) {
    // PATCH is naturally idempotent — safe to retry
    await netbox.patch(`/dcim/devices/${n.id}/`,
      { tenant, custom_fields: { cluster_id: n.cluster }});
  }
}

// Retry policy lives in config, not your code:
{ startToCloseTimeout: "5m",
  retry: { maximumAttempts: 5,
           backoffCoefficient: 2 } }
Rule of thumb: workflows decide, activities do. A failed activity retries with backoff automatically; a permanently failed one throws into the workflow, where you handle it like any exception.
// Pause for a human — durable for an hour or a month
const approved = defineSignal("approve");
const status   = defineQuery("status");

export async function bigOrderWorkflow(order) {
  let phase = "awaiting-approval";
  setHandler(status, () => phase);
  await condition(() => isApproved);   // sleeps durably
  phase = "provisioning";
  ...
}

# from anywhere, later:
temporal workflow signal --workflow-id ord-991 --name approve
Signals write into a running execution; queries read out without disturbing it. This is how "waiting on finance" becomes a first-class, crash-proof program state instead of a database limbo.
// The saga pattern = compensation as try/catch
const undo: Array<() => Promise<void>> = [];
try {
  const nodes = await claimNodes(8);
  undo.push(() => releaseNodes(nodes));
  await deployAll(nodes);
  undo.push(() => wipeAll(nodes));
  await verifyCluster();
} catch (err) {
  for (const step of undo.reverse()) await step();
  throw err;   // order fails clean: no half-clusters
}
Because the workflow itself is durable, the compensation is durable too — the undo runs even if the crash happens during the undo. Hand-rolled systems almost never get this part right.
The determinism contract (the one rule): workflow code must produce the same decisions on replay — so no direct clocks, random numbers, network calls, or global state inside workflows. Use the SDK's versions (workflow.now(), side-effect APIs) and put every real-world touch in an activity. Break this and replay diverges — the SDK will yell at you.

Things worth knowing before you lean on it

  • Determinism is a discipline, not a suggestion. Workflow code can't use clocks, randomness, I/O, iteration over unordered maps, or anything environment-dependent — the SDK provides safe equivalents. Teams internalize this in a week; code review enforces it forever.
  • Pass references, not payloads. Every input/output lands in the event history, and histories have size limits. Ship IDs and URLs through workflows; keep the 2 GB image in object storage. Very long-lived workflows use continue-as-new to reset history.
  • Activities run at-least-once — idempotency is on you. A retry after a timeout may re-execute a call that actually succeeded. PATCH-style operations, upserts, and idempotency keys make this a non-event; naive POSTs make it a double-charge.
  • Changing running code needs versioning. A workflow started last week replays against this week's deploy — the SDK's patching/versioning APIs (and worker deployment versioning) exist so old executions replay the old logic. Learn this before your first breaking change, not after.
  • The server is stateful infrastructure. Self-hosting means a Postgres-backed cluster (Helm chart exists) that you monitor, back up, and upgrade — its database joins your Postgres tier and your break-glass plan. Temporal Cloud trades money for that burden; note some multi-region features are Cloud-first.
  • It's an orchestrator, not a data plane. Don't stream metrics or bulk data through workflows; Temporal coordinates actions on other systems. High-throughput event flow belongs on a log (Kafka), with Temporal reacting to it.
  • Observability is half the product — use it. The Web UI's per-execution event history is the best debugger most teams have ever had for distributed processes; wire the server and worker metrics into your monitoring from day one (task-queue lag is your early-warning signal).

Run it yourself

LAB A — Dev server + first workflow

~20 min · needs the temporal CLI + Node or Go

  1. Start the all-in-one dev server (single binary, bundled UI, throwaway storage):

    brew install temporal        # or: curl -sSf https://temporal.download/cli.sh | sh
temporal server start-dev

    EXPECT: server on :7233, Web UI at http://localhost:8233.

  2. Run a sample. Clone the samples for your language (e.g. samples-typescript or samples-go), start the hello-world worker, then trigger the workflow per its README.

    EXPECT: in the UI, your execution with a step-by-step event history — Started, ActivityScheduled, Completed. That history IS the durability.

LAB B — Kill it and watch it not care

~30 min · builds on LAB A

  1. Write a 5-step toy saga: five activities that each log and sleep(10s), one durable timer between steps 3 and 4. Predict first: what will the history show if the worker dies at step 3?

  2. Start it, then kill the worker mid-run (Ctrl-C during step 3). Watch the UI: the execution stays Running. Restart the worker.

    EXPECT: execution resumes at step 4 — steps 1–3 are NOT re-executed (their results replay from history). This 30 seconds is the entire pitch, felt in your hands.

  3. Add a signal: make step 5 wait on an "approve" signal, then send it from a second terminal: temporal workflow signal --workflow-id <id> --name approve. Human-in-the-loop, demonstrated.

LAB C — Production shape

when it graduates

  1. Self-host for real: the official Helm chart onto your management cluster, persistence pointed at a Postgres in your database tier (own database, own backups). Workers become a normal Deployment; the Web UI goes behind your SSO front door. Wire server + worker metrics into your monitoring before the first real workflow ships.

Glossary & further reading

Workflow
The durable orchestration function: deterministic code whose execution is recorded step-by-step and can run for seconds or years.
Activity
A single unit of real-world work invoked from a workflow — API calls, DB writes, emails. Automatically retried; must be idempotent.
Worker
Your process (built with an SDK) that polls task queues and executes workflow and activity code — on your infra, with your credentials. The server never runs your code.
Event history
The append-only record of everything an execution has done — the source of truth from which state is rebuilt.
Replay
Re-running workflow code against its recorded history to reconstruct state after a crash or worker restart; completed steps return recorded results instead of re-executing.
Determinism
The contract that makes replay possible: workflow code must make identical decisions given the same history — no clocks, randomness, or I/O outside SDK APIs.
Task queue
The named channel matching work to worker fleets — also the lever for routing, rate control, and canary-deploying new worker versions.
Signal / Update
Asynchronous message into a running workflow (Signal) or a tracked, validated request that returns a result (Update) — how the outside world steers an execution.
Query
A synchronous, read-only peek at a running workflow's state ("percent complete?") without affecting it.
Saga / compensation
The pattern for multi-step transactions without distributed locks: on failure, run recorded undo steps in reverse. In Temporal, literally a try/catch.
Schedule
Cron-with-a-brain: starts workflows on a calendar with overlap policies, pause/resume, and backfill.
Namespace
Temporal's isolation unit — separate teams/apps get separate namespaces; Nexus provides durable calls across them.
Nexus
Durable service-to-service operations across namespace (and, in Cloud, region) boundaries behind explicit API contracts — service composition for durable executions.
Temporal Cloud
The managed server: you still run your own workers; Temporal runs the stateful brain. The buy-side of the self-host decision.
