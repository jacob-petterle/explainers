1

What it is

NetBox is a structured database — with a web UI and a full API — that models your physical and logical infrastructure. It merges two older tool categories into one data model:

Half oneDCIM

Where everything physically lives.

Data Center Infrastructure Management: the physical world as data.

  • SITES/RACKSBuildings, rooms, racks — down to which U each device occupies.
  • DEVICESEvery server, switch, PDU: manufacturer, model, serial, asset tag, role.
  • CABLESWhat's plugged into what — traceable end to end, port by port.
  • POWERFeeds, outlets, and draw, from utility panel to PSU.
Half twoIPAM

Who owns every address.

IP Address Management: the logical network as data.

  • PREFIXES/IPSEvery subnet and address — allocated, reserved, or free — with what it's assigned to.
  • VLANS/VRFSLayer-2 segments and routing domains, mapped to sites and racks.
  • BMC IPSIncluding the management addresses from Sheet 1 — NetBox is where "which IP is that iDRAC?" gets answered.
  • DNS/ASNNames, autonomous systems, and the glue records around them.
The one idea to keep: NetBox records intended state — what should exist. It has no agents, speaks no Redfish, and never probes your network. It is documentation with a schema: a wiki page that can't rot, because your automation reads it.
2

The job it solves

Every infrastructure team accumulates the same questions. Without a source of truth they're answered by spreadsheets, tribal memory, and walking to the rack:

"Which rack is that box in?"Device page → site, rack, U-position, and a rendered rack elevation.
"Is 10.4.2.17 free?"IPAM shows every address as allocated, reserved, deprecated — or available.
"What's on switch port 12?"Cable traces answer it without sending anyone with a flashlight.
"What does Ansible target?"Playbooks pull inventory from NetBox instead of a stale hosts file.
"Audit next week."Serials, asset tags, and locations export in one query.
"Does reality match the plan?"Compare what Redfish reports against what NetBox says should be there.
Design pattern, not just a tool: "network source of truth" means one authoritative dataset that humans edit and machines consume. If a fact about your infrastructure matters, it lives in NetBox; if NetBox and reality disagree, one of them is wrong — and now you know to look.

What NetBox is not

  • Not monitoring. It stores no live metrics, no health, no time series. "Is it up?" is Grafana's question; "should it exist, and where?" is NetBox's.
  • Not automation. It never reboots, configures, or deploys anything. It's the data automation reads — the noun, not the verb.
  • Not auto-magical. Its value equals its accuracy. Teams succeed with NetBox when updating it is part of the change process ("if it's not in NetBox, it doesn't exist"), and fail when it becomes another stale spreadsheet with better UI.

Run it yourself

LAB A — Click around the public demo

~10 min · just a browser

  1. Open demo.netbox.dev — a shared, pre-populated instance (login details are shown on its sign-in page). It resets periodically, so experiment freely.

  2. Trace one device end to end. Pick any device under Devices: note its rack elevation (a drawn picture of the rack), click an interface, follow its cable to the far switch port, then click its IP to land in IPAM. That round trip is the DCIM+IPAM data model.

    EXPECT: physical location → cabling → logical addressing, all cross-linked.

  3. Now pull the same data as JSON — everything the UI shows, the API serves:

    curl -s "https://demo.netbox.dev/api/dcim/devices/?limit=3" | jq '.results[] | {name, site: .site.name, rack: .rack.name}'

    EXPECT: the devices you just browsed, as machine-readable records — the shape Ansible's inventory plugin consumes.

LAB B — Run your own with Docker

~30 min · needs Docker

  1. Launch the community Docker stack from netbox-community/netbox-docker:

    git clone https://github.com/netbox-community/netbox-docker.git
cd netbox-docker
docker compose up -d

    First boot takes a few minutes; follow the repo's README to create your admin user, then open the printed local URL.

  2. Model a tiny lab: create one Site → one Rack → one Device (make up a server), give it an interface named iDRAC, and assign it a management IP. You've just recorded intent.

  3. Close the loop with Sheet 1's lab: if you ran the Redfish mockup or an OpenBMC QEMU, store its address in NetBox as the device's out-of-band IP. Then write five lines of Python: fetch the IP from NetBox's API, GET /redfish/v1/Systems from it, and compare the serial number to NetBox's record. Congratulations — you've built a drift check, the dashed blue line in FIG. 3-1.

Glossary & further reading

DCIM
Data Center Infrastructure Management — tracking the physical layer: sites, racks, devices, cables, power.
IPAM
IP Address Management — tracking the logical layer: prefixes, addresses, VLANs, VRFs, and what owns each.
Intended vs observed state
Intended = what should exist (NetBox). Observed = what actually answers (Redfish, monitoring). Healthy ops keep the two equal.
Drift
Any gap between intended and observed — a swapped part, an undocumented server, a reused IP. Drift found early is a ticket; found late, an outage.
VRF / VLAN
VLAN: a layer-2 network segment. VRF: an isolated routing table, letting the same IP ranges exist independently — both first-class NetBox objects.
Webhook
An HTTP call NetBox fires when data changes ("device added") so other systems react instantly instead of polling.
Out-of-band interface
The device's management port — the BMC connection from Sheet 1. NetBox models it as its own interface with its own IP.
Source of truth
The one dataset agreed to be authoritative. Everything else — dashboards, playbooks, docs — derives from it.
