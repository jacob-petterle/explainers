PROMETHEUS: THE HEARTBEAT RECORDER

The metrics engine of modern infrastructure: it goes out, asks every system "how are you?" on a schedule, remembers the answers as time series, and raises the alarm when they go wrong.

1

What it is

The modelpull

A database that goes and gets.

Prometheus inverts the usual flow: systems don't send it data — it scrapes them. Every target exposes a plain-text /metrics page; Prometheus visits each one on an interval (say, every 30s), timestamps what it reads, and appends it to its own embedded time-series database on local disk.

Things that can't speak /metrics natively get an exporter — a small translator that can (there's one for almost everything, including server BMCs).

The brainPromQL + rules

Metrics you can compute with.

Every sample is a name plus labels: temp_celsius{rack="B12",serial="7X2K9M3"}. PromQL slices, aggregates, and does math across those dimensions — and rules run PromQL on a schedule: recording rules precompute expensive queries, alerting rules fire alerts into Alertmanager, which routes pages to humans.

The relationship in one sentence: exporters expose, Prometheus collects and evaluates, Alertmanager tells humans, and dashboards (e.g. Grafana) ask it questions in PromQL.

2

The job it solves

Is everything up?Every scrape doubles as a health check — the built-in up metric is a liveness map of your fleet.
What's trending wrong?Temperatures creeping, disks filling, error rates rising — visible before they're incidents.
Page me when…Alert rules evaluate continuously, server-side, whether or not anyone is watching a dashboard.
Targets keep changingService discovery — Kubernetes-native or file-based — keeps the scrape list current without hand-editing.
Capacity planningWeeks of high-resolution history: growth rates, saturation math, and headroom in one query.
Agentless hardware viewA Redfish exporter polls BMCs — sensors from servers whose OS may not even be installed yet.
Why pull matters: scraping makes the monitoring system authoritative about what should exist. If a target vanishes, that's a fact Prometheus notices (up == 0) — with push systems, silence is ambiguous: dead, or just quiet?

Hands-on: scrape, query, alert, ship

# prometheus.yml — who to scrape, how often
scrape_configs:
  - job_name: redfish
    scrape_interval: 60s      # be gentle with BMCs
    file_sd_configs:
      - files: [ /etc/prom/targets/bmcs.json ]
    # bmcs.json is GENERATED from your inventory DB —
    # every target arrives pre-labeled:
    # {"targets":["10.99.4.107"],
    #  "labels":{"rack":"B12","tenant":"cust-royal"}}
file_sd is the bridge to a source of truth: a cron job renders the inventory system's device list into this JSON, and Prometheus reloads it automatically. Metrics are born knowing their rack and owner.
# The 4 PromQL moves that cover 90% of ops:

# 1 — instant state, filtered by labels
temp_celsius{rack="B12"}

# 2 — rate: counters are useless raw
rate(network_transmit_bytes_total[5m])

# 3 — aggregate across a dimension
avg by (rack) (temp_celsius)

# 4 — top offenders
topk(5, temp_celsius)

# liveness for free:
up{job="redfish"} == 0     # who's not answering?
Labels are the whole game: every aggregation, alert, and dashboard slice rides on them — which is also why careless labels become the #1 footgun (see Gotchas).
# rules/thermal.yml
groups:
- name: hardware
  rules:
  - alert: ChassisHot
    expr: avg_over_time(temp_celsius[10m]) > 45
    for: 10m              # flap protection
    labels:   { severity: page }
    annotations:
      summary: "{{ $labels.serial }} hot in {{ $labels.rack }}"
      runbook_url: https://wiki/runbooks/thermal
This evaluates every interval, forever, dashboards or no dashboards. Alertmanager then groups all ChassisHot alerts sharing a rack into one notification and routes by severity.
# Stream every sample to the long-term tier too
remote_write:
  - url: http://kafka-adapter:8080/receive
    # → Kafka topic → Iceberg tables (years of history)
  - url: https://mimir.internal/api/v1/push
    # → or a scale-out store on object storage

# v3 protocol note: Remote-Write 2.0 slashes
# payload size + CPU via string interning
This is the official answer to "Prometheus keeps weeks, I need years": keep the local TSDB lean for speed, and stream a copy to storage built for the long haul.

Things worth knowing before you lean on it

  • Cardinality is the #1 footgun. Every unique label combination is a separate series in memory. A label like customer_id is fine at 200 values; request_id will melt the server. Rule of thumb: labels are for dimensions you'd group by, never for unique identifiers.
  • Local storage is deliberately short-horizon. The embedded TSDB is built for days-to-weeks at high speed, not years. Don't fight it with giant retention — ship history out via remote_write and keep the hot server lean.
  • There is no external database — and that's not configurable. You cannot point Prometheus at Postgres/RDS. Its state is its own TSDB on local disk (a PersistentVolume in Kubernetes). Backup strategy = snapshots or, better, treating the long-term tier as the durable copy.
  • HA is charmingly primitive: run two identical Prometheis scraping the same targets; Alertmanager (clustered) deduplicates their alerts. It works — but the two servers' data will differ slightly; global consistency is what Mimir/Thanos are for.
  • Counters need rate(). Raw counter values are meaningless (they reset on restart). Nearly every "why is my graph a sawtooth?" question is a missing rate().
  • Be gentle with BMCs. Exporters that poll management controllers should scrape at 60s+, with timeouts — a BMC's tiny CPU wedges under aggressive polling, and a wedged BMC is a lost lifeline.
  • Pull needs reachability. Prometheus must reach every target's network (e.g. the isolated management VLAN for BMC exporters) — plan the monitoring server's network position deliberately. The Pushgateway exists for batch jobs only; using it as a general push door forfeits the liveness signal.
  • v2 → v3 has real (small) breaking changes. Prometheus 3.0 (Nov 2024) changed some flags, config, and scrape behaviors — read the migration guide; the payoff is a new UI, UTF-8 metric names, Remote-Write 2.0, and stabilized native histograms.

Run it yourself

LAB A — Prometheus watching itself

~10 min · needs Docker

  1. Run it. Out of the box it scrapes its own metrics:

    docker run -d -p 9090:9090 --name prom prom/prometheus

  2. Open http://localhost:9090 → Status → Targets.

    EXPECT: one target (itself), state UP — the pull model, visible.

  3. Query in the expression browser. Predict first: what should up return?

    up
rate(prometheus_http_requests_total[1m])
topk(3, prometheus_http_requests_total)

    EXPECT: up == 1 with job labels; a per-endpoint request rate; your first topk.

LAB B — A real target + a firing alert

~30 min · Docker

  1. Add node_exporter (real machine stats) and point Prometheus at it: run prom/node-exporter on port 9100, add a scrape_configs job for host.docker.internal:9100 in a mounted prometheus.yml, and restart. Setup details in the getting-started guide.

    EXPECT: two targets UP; try node_filesystem_avail_bytes and avg by (mode) (rate(node_cpu_seconds_total[1m])).

  2. Write an alert rule you can trip on purpose: e.g. node_load1 > 0.5 for: 1m in a rules file, then generate load and watch Alerts go pending → firing.

    EXPECT: the state machine live — and an intuition for what for: protects you from.

  3. Kill node_exporter and query up == 0. Absence as a signal, demonstrated.

LAB C — Hardware for real

when your fleet exists

  1. Deploy a Redfish exporter with reach into the management VLAN, generate its target file from your inventory system (labels included), scrape at 60s, and port the ChassisHot rule from Section 4. First dashboardable query: topk(10, temp_celsius) joined to owners in your dashboard layer.

Glossary & further reading

TSDB
Time-series database — storage optimized for timestamped samples. Prometheus embeds its own on local disk; no external database is used or supported.
Exporter
A small sidecar service that translates some system's internals (a BMC, a database, a switch) into a scrapeable /metrics page.
Labels / cardinality
Key-value dimensions on every metric (rack, serial). Each unique combination is its own stored series — cardinality is the count of those, and the resource that runs out first.
PromQL
Prometheus's query language: filter by labels, rate() counters, aggregate with by(), rank with topk(). basics
Scrape / target
One pull of one endpoint's metrics on an interval. The target list comes from static config or service discovery.
Service discovery / file_sd
Automatic target lists: Kubernetes-native watching, or a JSON file (file_sd) that anything — like an inventory database — can generate.
Recording & alerting rules
PromQL evaluated on a schedule: recording rules save results as new metrics; alerting rules emit alerts when an expression holds for a duration.
Alertmanager
The companion service that receives alerts and handles humans: grouping, deduplication, silences, and routing to Slack/pagers/email. docs
remote_write
A live stream of every ingested sample to another system — the doorway to long-term storage. Protocol 2.0 (v3 era) cuts payload size and CPU sharply.
Mimir / Thanos
Scale-out backends that accept remote_write (or sidecar uploads) and store years of metrics in S3-compatible object storage with a global query view.
Pushgateway
A narrow exception to pull: short-lived batch jobs push their final metrics here for Prometheus to scrape. Not a general ingestion door.
Native histograms
The v3-era histogram type: exponential buckets without pre-chosen boundaries — better resolution at lower cost, stable as of v3.8.
