What it is

What you touchUI

The dial you turn.

Auto mode is the setting in Cursor's model picker that hands a request to routing instead of a fixed model. It's existed for a while as a simple "pick something reasonable" mode — that's the part users already knew.

What actually runs itClassifier

The engine behind the dial.

Cursor Router, announced today, is a trained classifier that reads each request's query, context, task complexity, and domain, then assigns it to whichever model in Cursor's pool will do the job best for the money — and updates as new models ship.

The relationship in one sentence: Auto mode is the switch; Cursor Router is what actually flips depending on the request. Today's news is that the switch got a real engine behind it — and that engine is now reachable from code, via @cursor/sdk, not just the apps.

2

The job it solves

Cursor's own numbers frame the problem: about 60% of developers pick one model and stick with it for everything, from one-line edits to genuinely hard refactors. That means routine work quietly gets billed at frontier prices.

One model for everythingA single daily-driver model handles trivial edits at the same price as hard problems.
Spend outpaces outputAI spend grows faster than the value it produces when every request pays frontier rates.
Admins want a leverTeams need to cap cost without banning frontier models for the work that actually needs them.
Models ship monthlyHand-picking a favorite model goes stale fast as new frontier models arrive.
The core promise: frontier-quality output at roughly 30–60% lower cost, without anyone having to hand-pick a model per request.

What Cursor Router actually does

  • CLASSIFYScores each request on query, context, complexity, and domain before any model runs.
  • 3 MODESCost, Balance, Intelligence — you or your admin choose where to sit on the price/quality frontier.
  • CACHE-AWARETrained and evaluated with the cache-miss cost of switching models included, not hidden.
  • ADMINPer-team/group enablement, mode restrictions, default mode, model allow/block lists.
  • VISIBILITYThe routed model can be shown or hidden per request — hidden by default.
  • SDK-REACHABLEAvailable programmatically through @cursor/sdk, not only the desktop/web/iOS apps.

Things worth knowing

  • Teams/Enterprise only, for now. Cursor's own changelog says it's "on by default for Teams plans" and enabled by Enterprise admins — there's no mention of GA for individual free/Pro accounts yet.
  • "Savings" are relative, not free. Balance and Intelligence modes still bill at the routed model's own rate; the reported 30–60% savings are versus always calling the most expensive frontier model, not versus zero cost.
  • The routed model is hidden by default. If you need to know which model actually answered — for debugging or compliance — you (or your admin) have to turn on display, or check admin-side logs.
  • Trained on Cursor's aggregate traffic, not your codebase. The classifier learns from 600k+ live requests and broad A/B tests across "thousands of enterprise developers." Your mileage on unusual domains may differ from the published averages.
  • Switching models mid-conversation costs a cache miss. Cursor says its reported savings already account for this, but it's a real latency/cost hit worth knowing is happening under the hood.
  • The public SDK hadn't caught up to the branding as of this writing. The npm package (@cursor/sdk 1.0.24, published ~2 days before this launch) still exposes the mechanism under its older internal name (autoReview / classifier-backed Auto mode) rather than a first-class Router type — expect the typed API surface to catch up in a near-term release.
  • Try it yourself with zero risk — Section 9 walks through inspecting exactly what shipped in the SDK today, the same way this brief was researched.

Run it yourself

You don't need Teams/Enterprise access to verify what's actually in the public SDK today — you can inspect the exact same package this brief is based on.

LAB A — Flip Auto mode on

~2 min · needs a Cursor account

  1. Open the model picker in Cursor (desktop, web, or CLI) and select Auto instead of a named model.

  2. Choose an optimization mode — Cost, Balance, or Intelligence — and run a normal request.

    EXPECT: the request completes normally; on Teams/Enterprise with Router enabled, the model actually used may be visible depending on your admin's visibility setting.

LAB B — Inspect the shipped SDK yourself

~10 min · needs Node.js

  1. Check what version is live and pull it down without installing anything globally:

    npm view @cursor/sdk version
mkdir sdk-check && cd sdk-check
npm pack @cursor/sdk
tar xzf cursor-sdk-*.tgz

    EXPECT: a package/ folder containing dist/esm/*.d.ts type definitions.

  2. Grep for routing-related fields — this is exactly how the "SDK Types" tab in Section 4 was verified:

    grep -rn -i "router\|autoReview\|classifier\|ModelSelection" package/dist/esm/*.d.ts

    EXPECT: hits in options.d.ts (autoReview, ModelSelection, AgentDefinition) — the same excerpt shown above.

  3. Compare against the docs at cursor.com/docs/api/sdk/typescript to see if a first-class Router API has landed since this brief was written.

LAB C — Pilot it on a team

optional · needs Teams/Enterprise admin

  1. Enable Cursor Router for one group from the Cursor Dashboard, set the default mode to Balance, and leave visibility on so you can see what's being routed.

  2. Track cost per commit for a week and compare it against the published baseline numbers in Section 4 (Tab 3) for your own workload.

Glossary — the jargon, decoded

Cursor Router
Anysphere's intelligent model router, launched July 22, 2026: a classifier that assigns each Auto request to the model expected to perform best for the cost. announcement
Auto mode
The model-picker setting that delegates model choice to routing instead of a fixed model. Predates Cursor Router; now powered by it.
Classifier
A model trained to categorize input — here, trained on 600k+ real coding requests to predict which downstream model will satisfy the user best per request.
Optimization mode
Cost, Balance, or Intelligence — a dial that shifts Cursor Router along the cost/quality Pareto frontier.
Pareto frontier
The curve of "best possible quality for a given cost" options; moving along it trades one for the other rather than improving both at once.
Keep rate
How much agent-generated code remains in the codebase over time — one of two metrics (with user satisfaction) Cursor uses to judge routing quality.
@cursor/sdk
Cursor's TypeScript package for running Cursor's coding agent from your own code. npm
autoReview (SDK field)
A boolean on LocalAgentOptions that enables "classifier-backed Auto mode" — the SDK's current (pre-rebrand) name for Cursor Router's underlying mechanism.
ModelSelection (SDK type)
The typed shape ({ id, params? }) used to pin an AgentDefinition to a specific model, or set "inherit".
