METAL³: BARE METAL, THE KUBERNETES WAY

"Metal Kubed." Physical servers become Kubernetes resources: kubectl apply a YAML file, and a real machine in a real rack powers on, wipes itself, and installs your image.

1

What it is

The interfaceBMO + CRDs

kubectl for physical servers.

Metal³'s heart is the Baremetal Operator (BMO): a Kubernetes controller watching a custom resource called BareMetalHost. Each YAML object represents one physical machine — its BMC address, credentials, and the state you want it in.

Declare the desired state; the operator makes reality match. Hardware becomes GitOps-able: reviewed in pull requests, applied like any other manifest.

The engineIronic inside

A veteran provisioner in a new suit.

Metal³ doesn't reimplement provisioning — it runs OpenStack Ironic in pods and drives it. Ironic does the hard, decade-refined work: talking Redfish/IPMI to BMCs, network- or virtual-media-booting a discovery agent (IPA), wiping disks, writing images.

You never touch Ironic's API directly; the operator translates.

The relationship in one sentence: BMO is the Kubernetes face, Ironic is the muscle, the BMC is the hands — YAML in, provisioned metal out.

2

The job it solves: declarative metal

Everything else in a Kubernetes shop is declarative — deployments, networks, DNS. Physical servers were the last holdout, managed by scripts and tickets. Metal³ closes that gap:

New rack arrivesEnroll 20 machines by applying 20 YAML files. Inspection auto-discovers their CPUs, RAM, disks, NICs.
Cluster needs 5 more nodesScale a MachineDeployment from 10 → 15. Real servers power on and join.
Roll a new OS imageChange one image URL in a template; machines rebuild in controlled batches.
Node returned by a tenantDeprovisioning triggers automated disk wiping before the host re-enters the pool.
Audit what you ownkubectl get bmh -A — every machine, its state, and its hardware profile as data.
Hardware config driftFirmware/BIOS settings declared in CRDs get reconciled like everything else.

What the Baremetal Operator can do to a host

  • ENROLLRegister a machine from its BMC address + credentials (stored as a Kubernetes Secret).
  • INSPECTBoot the IPA agent from a ramdisk to auto-discover CPUs, memory, disks, and NICs into the resource's status.
  • PREPAREApply RAID, BIOS, and firmware settings declaratively before provisioning.
  • PROVISIONWrite any cloud-format image (qcow2 + cloud-init) to disk and boot it.
  • CLEANWipe disks on deprovision so hosts return to the pool safe for the next user.
  • POWERReboot, power on/off — expressed as fields on the resource, not commands you run.
  • CLUSTERVia CAPM3, hand hosts to Cluster API so whole Kubernetes clusters rise from raw metal.
Scope check: Metal³ manages the machine lifecycle — from raw metal to booted OS and back. It is not monitoring, not an inventory system of record, and not a general automation engine; it expects those to exist beside it.

Hands-on: the YAML that moves metal

# One physical machine = one resource (+ its BMC secret)
apiVersion: metal3.io/v1alpha1
kind: BareMetalHost
metadata: { name: node-07 }
spec:
  online: true
  bmc:
    address: redfish-virtualmedia://10.99.4.107/redfish/v1/Systems/1
    credentialsName: node-07-bmc-secret
  bootMACAddress: "a0:36:9f:12:34:56"
The bmc.address scheme picks the driver — redfish-virtualmedia:// boots the agent via the BMC with no DHCP infrastructure needed. Credentials live in a normal Kubernetes Secret.
# After enrollment, inspection fills status automatically
$ kubectl get bmh node-07 -o yaml
status:
  provisioning: { state: available }
  hardware:
    cpu:     { count: 64, arch: x86_64 }
    ramMebibytes: 524288
    storage: [ { name: /dev/nvme0n1, sizeBytes: 3840755982336 } ]
    nics:    [ { mac: a0:36:9f:12:34:56, model: ... } ]
You never typed any of this — the IPA agent booted on the real machine, looked around, and reported back. Your hardware inventory is now kubectl-queryable.
# Point the host at an image; the operator does the rest
spec:
  image:
    url:      http://images.local/ubuntu-24.04.qcow2
    checksum: http://images.local/ubuntu-24.04.qcow2.sha256
  userData: { name: node-07-cloud-init }  # first-boot config

# watch it happen:
$ kubectl get bmh -w
node-07   provisioning   ...
node-07   provisioned    ✓
Any cloud-format image works (qcow2 + cloud-init for first boot). Remove spec.image and the host deprovisions — wiping disks on the way back to the pool.
# One level up: whole clusters via Cluster API + CAPM3
kind: MachineDeployment          # like a Deployment, for machines
spec:
  replicas: 10
  strategy:
    rollingUpdate: { maxSurge: 1, maxUnavailable: 0 }
  template:
    spec:
      infrastructureRef: { kind: Metal3MachineTemplate, ... }
Change the template's image and machines rebuild one at a time — a rolling, gate-able rollout across physical hardware. Keep a small separate MachineDeployment as your canary pool: new images go there first.

The BareMetalHost lifecycle

Every host walks a strict state machine — the operator's whole job is moving hosts along it (and back):

S1REGISTERINGvalidating BMC address + credentials
S2INSPECTINGIPA ramdisk boots, auto-discovers hardware
S3PREPARINGRAID / BIOS / firmware settings applied
S4AVAILABLEin the pool, ready to claim
S5PROVISIONINGimage written to disk
S6PROVISIONEDrunning your OS; deprovision → clean → back to S4

There's also an externallyProvisioned escape hatch for machines that were set up by other means but should still be tracked and power-managed here.

Alternatives — the honest comparison

Metal³ is one of several serious bare-metal provisioners. It wins when Kubernetes is your control plane; it's overkill or awkward when it isn't:

OptionWhat it isStrengthsTradeoffs
Metal³metal3.ioK8s operator wrapping Ironic; hosts as CRDs; Cluster API integrationGitOps-native; declarative; clusters-from-metal via CAPI; battle-tested Ironic coreRequires a management K8s cluster first (bootstrap chicken-and-egg); Ironic's complexity is hidden, not gone
Ironic standalone+ Bifrost installerThe same engine, no Kubernetes — REST API + its own databaseNo K8s dependency; full Ironic feature surface; scriptable from anythingImperative REST, not declarative; you build the workflow layer yourself
MAASCanonical"Metal as a Service" — full-featured provisioner with polished web UI, DHCP/DNS/IPAM built inFastest path to a working bare-metal cloud; great UI; strong Ubuntu storyIts own world (not K8s-native); Canonical-centric; less natural fit for GitOps workflows
TinkerbellCNCF sandbox lineageMicroservices provisioning engine (originally Equinix Metal); workflows as templated actionsVery flexible workflow model; container-native; no Ironic baggageYou assemble more yourself; smaller ecosystem than Ironic's decade of hardware drivers
Foreman+ pluginsVeteran lifecycle manager (PXE, kickstart, config mgmt integration)Mature, huge install base, strong OS-config integration (Puppet/Ansible)Pre-cloud-native design; heavier to operate; weak K8s story
DIY: Ansible + Redfishplaybooks straight at BMCsNo provisioner at all — scripts drive power, virtual media, and images directlyZero new infrastructure; total control; fine for small static fleetsNo state machine, no cleaning guarantees, no pool concept — you reinvent all three at scale
  • PICK METAL³Kubernetes is (or will be) your control plane, you want GitOps for hardware, or you're building clusters-on-demand from a pool.
  • PICK IRONIC/BIFROSTYou want Ironic's power without running Kubernetes to get it — e.g. a control plane written in-house calling a REST API.
  • PICK MAASYou want a working bare-metal cloud this week, with a UI humans enjoy, and Ubuntu is home turf.
  • PICK TINKERBELLYou want maximal workflow flexibility and are comfortable assembling the pieces.
  • PICK DIYFleet under ~20 boxes, reimaged rarely — a provisioner would outweigh the problem.

Things worth knowing before you adopt it

  • The bootstrap chicken-and-egg is real. Metal³ runs on Kubernetes to provision the machines Kubernetes runs on. Standard answer: boot a temporary cluster (kind on a laptop, or one hand-installed node), provision the real management cluster, then clusterctl move ("pivot") the resources into it.
  • Choose your boot path early: virtual media vs iPXE. Drivers like redfish-virtualmedia:// boot the agent through the BMC — no DHCP servers to run, works across routed networks. iPXE-based drivers need DHCP/TFTP infrastructure on a provisioning network. Virtual media is the simpler operational story if your BMCs support it well.
  • BMC credentials become Kubernetes Secrets. Whoever can read those Secrets owns your hardware. Lock down RBAC on the namespace, and treat the management cluster itself as crown-jewel infrastructure.
  • Cleaning and inspection take real minutes. Every pool return means a reboot into the agent and a disk wipe; full-disk erasure on big NVMe can be slow. Budget for it in capacity planning — it's a security boundary between tenants, not a step to skip.
  • Homogeneous architectures only. x86_64 and ARM64 are both supported, but not mixed within one setup — plan fleets accordingly.
  • Images must be cloud-style. Provisioning writes a cloud-format image (e.g. qcow2) and hands first-boot config via cloud-init — golden-image workflows, not interactive installers.
  • Ironic is hidden, not gone. When provisioning fails at 3 AM, the logs you'll read are Ironic's. Budget learning time for its concepts (drivers, boot interfaces, cleaning steps) even though you interact through CRDs.

Run it yourself

LAB A — Touch the API (no hardware, 15 min)

needs Docker + kubectl + kind

  1. Boot a throwaway Kubernetes cluster with kind (Kubernetes-in-Docker):

    kind create cluster --name metal3-lab

  2. Install the BareMetalHost resource definitions straight from the Baremetal Operator repo:

    kubectl apply -k "github.com/metal3-io/baremetal-operator/config/crd?ref=main"

    EXPECT: baremetalhosts.metal3.io (and friends) created.

  3. Explore the shape of a physical server as an API object:

    kubectl explain baremetalhost.spec
kubectl explain baremetalhost.spec.bmc
kubectl explain baremetalhost.status.hardware

    EXPECT: the full schema — bmc.address, image.url, online, hardware inventory fields — documented like any built-in resource.

    No operator is running, so nothing reconciles — but you now know exactly what Metal³'s contract looks like, which is most of the mental model.

LAB B — A simulated datacenter (the real thing, fake metal)

~1–2 h · a Linux box with 16GB+ RAM & nested virt

  1. Run metal3-dev-env — the project's own environment that creates libvirt VMs posing as bare-metal servers, complete with emulated Redfish BMCs (via sushy-tools), then installs the full stack against them:

    git clone https://github.com/metal3-io/metal3-dev-env.git
cd metal3-dev-env
make   # provisions host deps, VMs, and the Metal3 stack

    Details and host requirements in the metal3-dev-env README and the Metal³ user guide.

  2. Watch a "machine" get provisioned end to end:

    kubectl get bmh -n metal3 -w

    EXPECT: hosts marching registering → inspecting → available — the Section 5 state machine, live — then provision one and watch a VM you can log into appear.

  3. Break something on purpose. Give a host wrong BMC credentials and watch the error surface in status; fix the Secret and watch it recover. Declarative reconciliation, felt directly.

LAB C — Real metal

optional · one server with a Redfish-capable BMC

  1. Point a BareMetalHost at a real machine: its BMC's redfish-virtualmedia:// address, credentials in a Secret (change the BMC's default password first — non-negotiable), and a qcow2 image on a local HTTP server. The YAML is identical to Lab B's; only the address is real. Isolate BMCs on a management network the cluster can reach, and keep them off the internet.

Glossary & further reading

BMC
Baseboard Management Controller — a small always-on computer on every server motherboard that can power, monitor, and reimage the host independently of its OS, even when the server is "off."
Redfish
The DMTF-standard HTTPS/JSON API that BMCs expose — one vendor-neutral way to command any server's hardware. dmtf.org
Ironic
OpenStack's bare-metal provisioning engine — a decade-refined state machine for booting, imaging, and cleaning physical servers. Metal³ runs it in pods and drives it. docs
IPA — Ironic Python Agent
A small ramdisk OS the target machine boots into during inspection and provisioning; it examines hardware, wipes disks, and writes images, reporting back to Ironic.
CRD / custom resource
Kubernetes' extension mechanism: define a new object type (like BareMetalHost) and it becomes a first-class API citizen with kubectl support.
Operator
A controller that watches custom resources and continuously reconciles reality to match their declared spec — the pattern BMO implements for servers.
BareMetalHost (BMH)
Metal³'s core resource: one object per physical machine, with desired state in spec and observed reality (including discovered hardware) in status.
Cluster API (CAPI)
Kubernetes' framework for managing whole clusters as resources — Machines, MachineDeployments — with pluggable infrastructure providers. book
CAPM3
The Cluster API Provider for Metal3 — the plugin that lets Cluster API claim BareMetalHosts, adding resources like Metal3Cluster and Metal3MachineTemplate.
Virtual media
A BMC feature that presents a remote ISO to the server as if it were a plugged-in USB drive — how Metal³ can boot its agent with zero DHCP infrastructure.
iPXE
Network booting: the machine's NIC fetches a boot image over the network. The alternative agent-boot path, requiring DHCP/TFTP on a provisioning network.
Cleaning
Automated disk erasure (and config reset) when a host is deprovisioned — the security boundary that makes pooled, multi-tenant metal safe.
Pivot (clusterctl move)
Migrating Cluster API + Metal³ resources from a temporary bootstrap cluster into the permanent management cluster it just provisioned — the answer to the chicken-and-egg.
Management cluster
The small Kubernetes cluster where BMO, Ironic, and CAPI run — the brain that provisions and manages all other machines and clusters.
