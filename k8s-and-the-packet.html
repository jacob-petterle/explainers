K8S & THE PACKET

Kubernetes never touches your traffic. It programs the machines that do. Once you see that, "does bare metal route through Kubernetes?" answers itself.

K8s isa control plane (a scheduler)
Packets flowNIC → switch → NIC, always
Who forwardsLinux kernel / NIC / switch ASIC
K8s' rolewrites the config they follow
Bare metalK8s not in path — at all
API down?traffic keeps flowing (Lab B)
1

The two planes

Control planeDECIDES

The clipboard: decides, never carries.

A control plane is software that holds desired state ("pod X should exist on node 3"; "tenant A owns VLAN 100") and pushes configuration to make it true. The Kubernetes API server, its controllers, a CNI's agent, and fabric managers like Netris or OVN are all control planes.

Control planes talk in API calls and config writes. They are slow-path by design: involved when something is created, moved, or deleted — then they get out of the way.

Data planeCARRIES

The conveyor: carries, never decides.

The data plane is what actually moves bytes: the Linux kernel's routing tables and eBPF/iptables programs, the NIC, and the switch ASIC. It executes, per-packet, whatever rules the control plane installed — at nanosecond speeds no API call could ever match.

Crucially, the data plane keeps running if the control plane dies. Rules already installed keep forwarding. You'll prove this yourself in Lab B.

The relationship in one sentence: the control plane programs the path; it is never on the path. "Traffic routes through Kubernetes" is a category error — like saying your commute routes through the city's traffic-light control room.

2

The questions this untangles

Bare-metal tenant traffic — through K8s?Never. It goes tenant NIC → switch fabric → destination. If the operator runs K8s internally (to host provisioning software), that cluster configured the switches once — it doesn't see packets.
What's a "Kubernetes VPC" then?Marketing shorthand. A VPC is isolation config (VLANs/VXLAN, routes, ACLs) pushed into switches and hosts by some control plane. Nothing called "a VPC" ever forwards a packet.
API server crashes — do pods go dark?No. Existing pods keep talking; installed kernel/switch rules keep forwarding. You just can't change anything until it's back.
How does a Service IP work?It doesn't "go somewhere." Every node's kernel carries a rule: "dst 10.96.0.7 → rewrite to one of these pod IPs." The rewrite happens on the node where the packet already is.
Why do GPU pods grab the NIC directly?Overlay networking costs encapsulation and kernel hops. RDMA pods use SR-IOV/host-device via Multus — the fastest K8s networking is K8s stepping aside.
Then what IS K8s doing all day?Reconciling: watching desired vs actual state and issuing config. A scheduler and a very persistent janitor — not a router.
The core insight: every networking layer has this same split. K8s : kernel rules :: Netris : switch config :: OVN : flow tables :: UFM : InfiniBand fabric. Learn the split once and every "does X route through Y?" question becomes: is Y installing rules, or executing them?

What each plane concretely does

  • CONTROLAPI server stores objects; controllers reconcile; kubelet starts containers; CNI plugin allocates the pod IP and installs routes/eBPF. All slow-path.
  • DATAKernel matches each packet against routes, conntrack, eBPF programs; NIC DMAs it; switch ASIC forwards it by MAC/VLAN. All fast-path.
  • HANDOFFThe only connection between the planes is installed configuration — there is no wire from the API server to your packets.
  • BARE METALFor a bare-metal tenant node, K8s' involvement ended at provisioning (BMC/Redfish calls, switch-port VLAN assignment). Steady-state: zero.

Hands-on mechanics — where the rules actually live

Four exhibits, all inspectable on any cluster (Lab A builds you one). Each is a piece of installed configuration — the only artifact the control plane ever leaves behind in the data path.

# On any K8s node: pod networking is just... the Linux routing table.
$ ip route
10.244.0.0/24 via 172.18.0.3 dev eth0    ← "pods on node A live behind node A"
10.244.1.0/24 via 172.18.0.4 dev eth0    ← installed by the CNI at join time
10.244.2.2 dev veth1a2b scope host        ← a local pod's veth pair

# The pod's end of the wire:
$ kubectl exec mypod -- ip addr    # eth0 here = veth peer in the node kernel
Takeaway: there is no "Kubernetes network device." A pod's veth plugs into the node's kernel; the kernel forwards using ordinary routes the CNI wrote. Delete the CNI pod and traffic still flows — the routes remain.
# A Service "IP" (ClusterIP) is a rewrite rule on EVERY node, not a place:
$ iptables -t nat -L KUBE-SERVICES | grep my-svc
KUBE-SVC-X7... tcp dpt:80 10.96.14.7       ← "if dst is the Service IP..."
$ iptables -t nat -L KUBE-SVC-X7...
KUBE-SEP-A1... statistic mode random probability 0.5   ← "...pick a backend pod"
KUBE-SEP-B2...                                      and DNAT to its real IP

# (Cilium does the same job as an eBPF map instead of iptables chains)
Takeaway: when you curl a Service, kube-proxy isn't proxying anything — it already programmed your node's kernel to rewrite the destination. The packet is corrected where it already is, then travels node-to-node like any other.
# For GPU/RDMA traffic, even the kernel is too much middleman.
# Multus attaches a SECOND, direct interface to the pod:
apiVersion: v1
kind: Pod
metadata:
  annotations:
    k8s.v1.cni.cncf.io/networks: sriov-rdma-net   ← a Virtual Function of the
spec:                                                 real NIC, passed into the pod
  containers:
  - resources:
      limits:
        nvidia.com/gpu: 8
        rdma/rdma_shared_device_a: 1

# NCCL then talks straight to the NIC hardware. CNI overlay: bypassed.
Takeaway: the highest-performance Kubernetes networking is Kubernetes arranging its own absence. SR-IOV/host-device hands the pod real NIC hardware; the east-west RDMA fabric (see companion sheet "RoCE & the Switch") never touches the overlay.
# "Create a VPC" — what actually happens, in any stack:

tenant intent      "tenant A: isolated network 10.1.0.0/16"
      ↓ API call
control plane      Netris / OVN / Neutron / your thin API
      ↓ config writes (the ONLY thing that moves)
switch fabric      VLAN 100 / VXLAN VNI 10100 on tenant A's ports
host kernels       VRF / routes / ACL rules
DPUs (if any)      per-tenant ACLs on the NIC's own cores

# Result: a "VPC" = rows in a database + config in devices.
# No packet ever visits a thing called VPC.
Takeaway: AWS invented "VPC" as a product name for exactly this: distributed isolation config with a nice API. When a vendor says their platform "provides VPCs for bare metal," ask one question: which device executes the isolation, and who programs it?

Alternatives & comparables — ways a pod can reach the wire

Inside Kubernetes, "networking" is a pluggable choice. These are the four serious data-path options for a GPU cloud; most real clusters combine two or three.

  • PICK OVERLAYControl-plane clusters, internal tooling, anything where simplicity beats the last 20% of throughput.
  • PICK CILIUMThe general-purpose default in 2026: eBPF speed, NetworkPolicy, Hubble observability — without re-architecting the fabric.
  • PICK UNDERLAYMulti-tenant KaaS where tenant pods need fabric-enforced isolation mapped to real VLANs/VXLANs (Kube-OVN is the OSS workhorse here).
  • PICK SR-IOVAlways — for the GPU/RDMA interface specifically. It coexists with any of the above via Multus: overlay for pod chatter, direct NIC for NCCL.

Where the split shows up

Bare-metal-as-a-service

The operator's provisioning stack (often itself on K8s) drives BMCs over Redfish and assigns switch ports to tenant VLANs — pure control-plane work. The tenant's packets then flow NIC-to-switch with the operator's software nowhere in the path.

Metal³ · Ironic · NVIDIA NICo · Netris

Kubernetes-as-a-service

Virtual control planes (one API server per tenant) share physical nodes; the data path is whatever CNI + fabric config the operator installed. Tenant isolation quality = fabric config quality, not K8s namespace boundaries.

vCluster · Kube-OVN · Cilium

GPU training/inference pods

Pods get two personalities: an ordinary CNI interface for control chatter, and a Multus-attached SR-IOV interface where NCCL/RDMA runs at wire speed. Two planes, visible inside one pod spec.

Multus · SR-IOV operator · NVIDIA Network Operator

Load balancers on bare metal

No cloud LB exists, so MetalLB answers ARP for a Service's external IP (L2 mode) or announces it via BGP — again: a controller that programs the network's existing mechanisms rather than sitting in the path (in L2 mode, traffic does land on one chosen node first).

MetalLB · kube-vip · BGP

Ecosystem — decoding the names

Inside the node

  • kubeletThe node agent: starts containers, calls the CNI plugin. Control plane's hands.
  • CNIContainer Network Interface — the plugin contract: "pod created, give it networking."
  • kube-proxyMisnamed: not a proxy. Programs Service rewrite rules (iptables/IPVS) into the kernel.
  • eBPFSandboxed programs the kernel runs per-packet — how Cilium replaces iptables.

CNI implementations

  • CiliumeBPF-based, CNCF-graduated; the mainstream high-performance choice.
  • CalicoRouting-centric (BGP-capable) with a policy engine; overlay optional.
  • Kube-OVNBrings OVN's VPC/VLAN model into K8s — the OSS route to fabric-real tenant isolation.
  • FlannelThe minimal VXLAN overlay; fine when networking isn't the product.

At the fabric edge

  • MultusLets a pod have multiple interfaces — the door to SR-IOV and RDMA.
  • SR-IOVOne physical NIC presents many Virtual Functions; pods/VMs get real hardware queues.
  • MetalLBLoadBalancer Services on bare metal via ARP or BGP announcements.
  • NVIDIA Network OperatorInstalls the RDMA/SR-IOV plumbing (drivers, device plugins) cluster-wide.

Gotchas — security first

  1. Namespaces are not walls. K8s namespaces and NetworkPolicies are control-plane constructs enforced by software the node runs. For hostile multi-tenancy, security auditors (and GPU-cloud rating rubrics) demand isolation the tenant can't reach even with a container escape: separate nodes, fabric VLANs, DPU ACLs. Don't sell namespace isolation as tenant isolation.
  2. hostNetwork pods skip everything. A pod with hostNetwork: true uses the node's interfaces directly — no CNI, no NetworkPolicy. Fine for infra agents; catastrophic if tenant workloads can request it. Gate it with admission policy.
  3. The overlay MTU trap. VXLAN steals ~50 bytes per packet. If pod MTU isn't reduced to match, large packets silently fragment or drop, and only some traffic breaks. If big transfers hang but pings work, check MTU first.
  4. Control-plane outage ≠ outage — but don't relax. Existing traffic survives an API-server crash (Lab B), but nothing can be scheduled, healed, or scaled. Conversely: don't debug a traffic problem by staring at the API server — it was never in the path. Wrong plane, wrong tools.
  5. Service IPs don't exist off-cluster. ClusterIPs are rewrite rules inside member nodes' kernels. A bare-metal tenant (or your laptop) can't reach one — that's what LoadBalancer/MetalLB/ingress are for. "Why can't the BM node hit 10.96.x.x" is this, every time.
  6. Two planes, two failure smells. Config that won't apply, pods stuck Pending, "unable to connect to the server" = control plane. Drops, latency, one-direction traffic, MTU weirdness = data plane. Diagnosing the right plane first cuts most network debugging in half.
  7. Trust, then verify — in the labs below. Every claim here is checkable in 20 minutes with a free local cluster.

Run it yourself

LAB A — X-ray a cluster's data path

~15 min · needs Docker · kind

Build a throwaway two-node cluster and go find the "Kubernetes network" — you'll discover it's routes, veths, and iptables all the way down.

  1. Install kind (Kubernetes-in-Docker) and create a 2-node cluster.

    curl -Lo ./kind https://kind.sigs.k8s.io/dl/latest/kind-linux-amd64
chmod +x ./kind && sudo mv ./kind /usr/local/bin/
cat <<EOF | kind create cluster --config=-
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes: [{role: control-plane}, {role: worker}]
EOF

    EXPECT: two Docker containers acting as nodes; `kubectl get nodes` shows both Ready. (kubectl: sudo apt install -y kubectl or see kind's docs.)

  2. Run two pods and inspect the "network" from inside a node.

    kubectl create deploy web --image=nginx --replicas=2
kubectl get pods -o wide                    # note pod IPs + nodes
docker exec -it kind-worker ip route        # the node's kernel routing table
docker exec -it kind-worker ip addr | grep veth

    EXPECT: routes for each node's pod CIDR + one veth per local pod. That routing table IS pod networking — installed by the CNI, executed by the kernel.

  3. Find a Service IP living in iptables.

    kubectl expose deploy web --port=80
kubectl get svc web                         # note the ClusterIP
docker exec -it kind-worker iptables-save | grep -A2 "default/web"

    EXPECT: NAT rules matching the ClusterIP and DNAT-ing to the two pod IPs — the SERVICE IP tab of Section 4, live.

LAB B — Kill the control plane; traffic shrugs

~10 min · continues Lab A

The claim from Section 1, proven: the data plane doesn't need the control plane to keep forwarding.

  1. Start a continuous pod-to-pod ping that logs every packet.

    POD_IP=$(kubectl get pod -l app=web -o jsonpath='{.items[0].status.podIP}')
kubectl run pinger --image=busybox --restart=Never -- sh -c "ping $POD_IP"
sleep 5 && kubectl logs pinger | tail -3    # confirm it's flowing

    EXPECT: steady icmp_seq replies between two pods across the cluster.

  2. Now pause the entire control plane.

    docker pause kind-control-plane
kubectl get pods                            # FAILS — API server unreachable
sleep 30

    EXPECT: kubectl errors ("connection refused") — Kubernetes is, for all management purposes, dead. Wait the 30 seconds.

  3. Resurrect it and read the ping log.

    docker unpause kind-control-plane
sleep 10
kubectl logs pinger | awk -F'icmp_seq=' '{print $2}' | cut -d' ' -f1 | sort -n | awk 'NR>1 && $1!=prev+1 {print "GAP after", prev} {prev=$1}'

    EXPECT: no GAP lines — the sequence numbers are continuous. Packets flowed the entire time Kubernetes was frozen, because the kernel rules were already installed. The control plane programs the path; it is not on it. (Cleanup: kind delete cluster)

LAB C — Graduation: give a pod a real second NIC

~30 min · continues Lab A · Multus

  1. Install Multus, the meta-CNI that lets pods have multiple interfaces:

    kubectl apply -f https://raw.githubusercontent.com/k8snetworkplumbingwg/multus-cni/master/deployments/multus-daemonset.yml

    EXPECT: kube-multus pods Running in kube-system.

  2. Define a second network (macvlan on the node's real interface — the stand-in for SR-IOV on real hardware):

    cat <<EOF | kubectl apply -f -
apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata: {name: direct-net}
spec:
  config: '{"cniVersion":"0.3.1","type":"macvlan","master":"eth0",
            "mode":"bridge","ipam":{"type":"host-local",
            "subnet":"172.18.100.0/24"}}'
EOF
kubectl run twonic --image=busybox --restart=Never \
  --annotations=k8s.v1.cni.cncf.io/networks=direct-net -- sleep 3600
kubectl exec twonic -- ip addr

    EXPECT: the pod shows eth0 (CNI overlay) AND net1 (direct attachment) — the exact dual-personality pattern GPU pods use, with net1 standing in for the RDMA-capable Virtual Function.

  3. On real gear, the graduation step is the same annotation pointing at an SR-IOV network via the SR-IOV operator — then Lab B of the companion sheet "RoCE & the Switch" (rping/ib_send_bw) runs inside the pod over net1.

    EXPECT: RDMA at wire speed from a pod, CNI overlay untouched. Both sheets, one demo.

Glossary — the jargon, decoded

Control plane
The software that decides and configures: holds desired state, reconciles reality toward it, and writes rules into devices. Slow-path; not on the wire.
Data plane
The machinery that forwards each packet by executing installed rules: kernel, NIC, switch ASIC. Fast-path; survives control-plane outages.
CNI
Container Network Interface — the plugin standard kubelet calls when a pod is created ("wire this pod up"). Cilium, Calico, Kube-OVN, and Flannel are CNI implementations. cni.dev
veth pair
A virtual patch cable with two ends: one becomes the pod's eth0, the other sits in the node's kernel. How every non-hostNetwork pod reaches the world.
kubelet
The per-node agent of the control plane: watches the API for pods assigned to its node, starts containers, invokes the CNI plugin.
kube-proxy
Despite the name, forwards nothing. Programs each node's kernel (iptables or IPVS) with Service-IP rewrite rules. Cilium can replace it with eBPF.
conntrack
The kernel's connection-tracking table — how a DNAT'd Service packet's replies get un-rewritten on the way back.
eBPF
Safe, JIT-compiled programs attached inside the running kernel — per-packet logic without kernel modules. The engine under Cilium. ebpf.io
Overlay / underlay
Underlay: the physical switched/routed network. Overlay: a virtual network tunneled on top of it (e.g. pod VXLAN). Overlays trade a small encapsulation tax for independence from the underlay's design.
VXLAN
The standard overlay encapsulation: an L2 frame carried in a UDP packet with a 24-bit network ID. Used both by CNIs (pod overlays) and by fabrics (tenant isolation at scale).
Service / ClusterIP
A stable virtual IP for a set of pods — implemented as a load-balancing rewrite rule present on every node in the cluster, and meaningless outside it.
NetworkPolicy
K8s' firewall object: allow/deny rules between pods, enforced by the CNI in software on each node. Useful hygiene; not hostile-tenant isolation.
hostNetwork
A pod setting that skips the CNI entirely and uses the node's own interfaces — outside every NetworkPolicy. For infrastructure agents only.
Multus
A "meta-CNI" that attaches additional interfaces to a pod beyond the default one — the mechanism for giving GPU pods a direct RDMA-capable NIC. github
SR-IOV
Single-Root I/O Virtualization: one physical NIC exposes many Virtual Functions, each assignable to a pod or VM as real hardware — near-native performance, no kernel middleman.
MetalLB
Gives bare-metal clusters working LoadBalancer Services by answering ARP for the external IP (L2 mode) or announcing it over BGP. metallb.io
