The open-source glass over your infrastructure: it doesn't store your data — it queries every system where it lives and composes the answers into dashboards, alerts, and (yes) custom UIs.

1

What it is

The modelfederation

A window, not a warehouse.

Grafana holds almost no data. Each dashboard panel sends a live query — PromQL to Prometheus, SQL to Postgres, LogQL to Loki — through a data source plugin, and renders whatever comes back.

That's the superpower: your metrics, your inventory database, your logs, and your business tables can all appear on one screen without ever being copied into a second system.

The scopeplatform

A platform wearing a dashboard costume.

Beyond charts: a full alerting engine (rules, routing, silences), user auth and team permissions, templated variables, dashboards-as-code with Git workflows, and a plugin SDK that goes all the way to entire custom applications mounted inside Grafana.

The relationship in one sentence: your databases keep the truth; Grafana is where humans look at it — and increasingly, where they act on it.

2

The job it solves

Real infrastructure truth is scattered: live metrics in one store, intended state in another, logs in a third, business context in a fourth. Grafana's job is making that scatter visually coherent without forcing a data migration:

On-call at 3 AMOne dashboard: the alerting node, its temperature history, its neighbors, its logs — triage without tab-hopping.
"Which customer is affected?"Join live metrics against your inventory/business tables in the same panel.
The NOC wallKiosk-mode dashboards cycling fleet health on a TV — the classic datacenter heartbeat display.
Exec wants a viewSame data, different dashboard: capacity, growth, and cost trends with none of the debugging noise.
Alert → runbookAlert rules carry links and context, and fire into Slack/pager routes with grouping and silences.
Dashboards drift & sprawlVersion them in Git, review changes as pull requests, provision them as code.
What it deliberately is not: a database, a collector, or a storage engine. Grafana pairs with things that store (Prometheus, Postgres/Timescale, Loki) and things that collect (exporters, agents). It's the presentation and decision layer.

Hands-on: query, join, alert, code

# A panel is: one query + one visualization.
# PromQL — hottest 5 servers by inlet temperature:
topk(5,
  redfish_thermal_sensor_celsius{sensor="Inlet"}
)

# templated with a dashboard variable:
redfish_thermal_sensor_celsius{rack="$rack"}
Variables ($rack, $tenant) turn one dashboard into a hundred: a dropdown re-scopes every panel at once.
# Two queries in ONE panel (mixed data sources):
A (Prometheus):  avg by (serial)
                 (redfish_thermal_sensor_celsius)
B (Postgres/NetBox, read-only):
   SELECT serial, rack, position, tenant
   FROM   netbox_view_devices;

# Transformation: "Join by field" on `serial`
# → one table: temp · rack · position · tenant

# v12+: SQL Expressions — write the join AS SQL
SELECT a.serial, a.value AS temp_c, b.rack, b.tenant
FROM   A a JOIN B b ON a.serial = b.serial
ORDER  BY temp_c DESC
This is the answer to "can I join my NetBox data?" — live metrics from Prometheus, intent from NetBox's database, merged at view time. No pipeline, no copy. Keep result sets small; this join runs per panel refresh.
# Alert rule (concept): fire when a chassis runs hot
condition:  avg_over_time(inlet_celsius[10m]) > 45
for:        10m
labels:     { severity: page, rack: {{ $rack }} }
annotations:
  summary:     "{{ $serial }} hot in {{ $rack }}"
  runbook_url: https://wiki/runbooks/thermal

# routing: severity=page → PagerDuty; else → Slack
Grafana ships a full alerting engine (rules, routing, grouping, silences). Teams already running Prometheus's Alertmanager can keep it — Grafana speaks to both worlds.
# Dashboards are JSON — treat them like code:
# provisioning/dashboards/fleet.yaml
providers:
  - name: fleet
    options: { path: /etc/grafana/dashboards }

# or Terraform:
resource "grafana_dashboard" "fleet" {
  config_json = file("fleet.json")
}

# or Git Sync (v12): dashboards ↔ GitHub, PRs and all
Version, review, and deploy dashboards like software — the cure for dashboard sprawl and "who changed the NOC wall?"

Custom UIs — how far can you push it?

"Does Grafana support custom UIs?" Yes — it's a ladder, and each rung costs more engineering but buys more control:

  • Dashboards + variables + drilldowns. Dropdowns, links between dashboards, conditional show/hide, tabs. Most "we need a custom app" requests die happily here.COST: hours · SKILL: point-and-click
  • Canvas panel. Free-form layouts: place metric-bound elements on a background image — a rack elevation with live temperature badges on each U position is the classic datacenter move.COST: hours · SKILL: point-and-click + patience
  • Community "business" panels. Plugins like the Volkov Labs suite add HTML/handlebars panels, dynamic forms (write back to APIs!), and buttons — custom-ish UI without writing a plugin.COST: days · SKILL: HTML/templates
  • Your own panel plugin. A React component receiving query results — any visualization you can imagine, living inside normal dashboards.COST: days–weeks · SKILL: React + plugin SDK
  • A full app plugin. Entire custom pages with their own routes, navigation, and backend, mounted inside Grafana — using its auth, its data sources, its Scenes framework for dashboard-like behavior. This is "build my own ops portal without building auth, RBAC, and 40 data connectors."COST: weeks · SKILL: React/Go + create-plugin toolkit

The three ways to get NetBox data in

  • SQL DIRECTAdd NetBox's PostgreSQL as a data source (read-only user, ideally a replica) and query inventory with plain SQL — then join against metrics as in Section 4. Most powerful, zero copies.
  • REST / JSONCommunity data-source plugins (e.g. Infinity / JSON API) query NetBox's REST API directly — no DB access needed, respects NetBox permissions, slightly clunkier for big joins.
  • LABEL ENRICHUpstream: generate Prometheus scrape targets from NetBox so every metric already carries rack/tenant labels — then no view-time join is needed at all. Best for alerts, which want labels, not joins.

Things worth knowing before you lean on it

  • View-time joins are for humans, not pipelines. Transformations and SQL Expressions run per panel refresh, in Grafana — great for a 50-row "hot nodes with owners" table, wrong for joining millions of rows. Heavy correlation belongs in your warehouse (Timescale/Iceberg); Grafana then just displays the result.
  • Protect the sources you point it at. Every dashboard viewer triggers real queries. For NetBox: a read-only Postgres role (or replica) — never the app's own credentials. For Prometheus: watch expensive PromQL on high-refresh NOC screens.
  • Know your license and edition. Grafana OSS is AGPLv3 — fine to run internally; embedding it inside a commercial product or offering it as a service demands legal attention (that's what Enterprise/Cloud licensing exists for). Some features (fine-grained RBAC, SCIM, white-labeling) are gated beyond OSS.
  • Two alerting worlds exist. Grafana's built-in alerting and Prometheus's Alertmanager overlap. Pick one as the system of record for pages; running both halfway is how alerts get lost.
  • Dashboard sprawl is the default failure mode. Without as-code discipline, you get 400 near-duplicate dashboards nobody trusts. Provision the important ones; treat click-built dashboards as drafts.
  • Grafana sees your data — secure it like it matters. It holds credentials to every store it queries. SSO, least-privilege data source accounts, and no public exposure without auth in front.
  • Plugins vary in maintenance. Community data sources (JSON/Infinity, business panels) are load-bearing for the custom-UI story — check signing, activity, and pin versions.

Run it yourself

LAB A — Zero install: the public playground

~10 min · just a browser

  1. Open play.grafana.org — Grafana Labs' public sandbox, pre-loaded with data sources and demo dashboards. No account needed.

  2. Open any dashboard, then a panel's Edit view. Find the three stages from FIG. 3-1: the query, the Transformations tab, the visualization picker. Change the viz type and watch the same data re-render.

    EXPECT: the mental model clicks — a panel is query → transform → picture, nothing more.

  3. Try a transformation: add "Join by field" or "Organize fields" to a multi-query panel and watch two result sets become one table — the Section 4 join, hands-on.

LAB B — Your own instance + a real cross-source join

~30 min · needs Docker

  1. Run Grafana:

    docker run -d -p 3000:3000 --name grafana grafana/grafana-oss

    Log in at http://localhost:3000 (admin/admin, then change it). Add the built-in TestData data source and build one panel to learn the editor.

  2. Add a SQL source and join. Point a PostgreSQL data source at any Postgres you have (the NetBox instance from a netbox-docker setup works perfectly with a read-only user). In one panel: query A from TestData (fake metrics), query B from Postgres (inventory rows), then Transformations → Join by field on a shared column.

    EXPECT: one table mixing "live" numbers with database context — the metrics-meet-inventory pattern in miniature.

  3. Make it alert: create an alert rule on your TestData series with a threshold, add a runbook_url annotation, and wire a contact point (a test webhook or email). Fire it by lowering the threshold. You've now touched every major subsystem.

LAB C — The real stack

when your fleet exists

  1. Wire it for real: Prometheus + a Redfish/IPMI exporter as the metrics source, NetBox's Postgres (read-only role) as the inventory source, and — the pro move — generate Prometheus's scrape targets from NetBox so every metric is born carrying rack and tenant labels. Your first dashboard: fleet temperatures, joined to owners, with one alert rule paging on sustained heat.

Glossary & further reading

Data source plugin
Grafana's adapter to one kind of store (Prometheus, Postgres, Loki, REST APIs…). Hundreds exist; each brings its own query editor.
PromQL
Prometheus's query language for time-series metrics — the dialect behind most infrastructure panels. docs
Loki / LGTM
Grafana Labs' log store (queried with LogQL); LGTM = Loki, Grafana, Tempo (traces), Mimir (metrics at scale) — the vendor's full open-source stack.
Transformation
A post-query step inside a panel: join by field, filter, organize, compute — where results from different data sources get merged at view time.
SQL Expressions
A v12 feature: write SQL that treats your panel's query results (from any sources) as tables and joins them — view-time federation with a real JOIN syntax.
Variables / templating
Dashboard-level dropdowns ($rack, $tenant) substituted into every query — one dashboard, many scopes.
Unified alerting
Grafana's built-in rule engine: conditions evaluated on queries, labeled and routed to contact points, with grouping and silences.
Panel plugin
A custom visualization: a React component receiving query results, installable into any dashboard.
App plugin
The top of the customization ladder: whole custom pages with routes and optional backend, mounted inside Grafana and reusing its auth and data sources. plugin tools
Scenes
Grafana's frontend framework for building dashboard-like experiences (time ranges, variables, panels) inside app plugins.
Provisioning / Git Sync
Managing dashboards, data sources, and alerts as files — via config, Terraform, or (v12) synced directly with a Git repository, PRs included.
Kiosk mode
Chrome-free full-screen display mode, usually with a playlist rotating dashboards — the NOC-TV feature.
AGPLv3
Grafana OSS's license: free to run and modify, but offering it as a network service triggers source-sharing obligations — the reason embedded/commercial uses go through Enterprise/Cloud licensing.
NetBox
An open-source infrastructure source of truth (racks, devices, IPs) backed by PostgreSQL — which is exactly why Grafana can query and join it directly.
