1

What it is — the engine vs. the business

Open source · Apache 2.0vcluster-oss

A Kubernetes cluster running inside a pod.

vCluster runs a real, CNCF-conformant Kubernetes API server + controller manager per tenant as a pod on your host cluster, backed by SQLite (via kine) or your own etcd. A syncer copies the few objects that must be real — pods, services, PVCs — down into a host namespace, where the host's nodes actually run them.

Tenants get their own API server, CRDs, RBAC, and versions: it feels like a dedicated cluster, at the cost of one pod, in seconds.

Commercial · license-gatedvcluster-pro + platform

Everything that makes it a sellable cloud product.

vCluster Labs (formerly Loft Labs) monetizes the layer around the engine: the Platform (UI, tenant self-service, SSO, quotas, sleep mode, audit), plus gated engine features — Private Nodes, embedded etcd, hybrid scheduling, DRA sync, Auto Nodes — enforced by license checks compiled into the default image.

Since Jan 2026 there's a middle "Free" tier: formerly-Enterprise features at $0, but license-validated and hard-capped at 64 vCPUs or 32 GPUs — four H100 boxes.

The relationship in one sentence: the OSS engine gives every tenant a control plane; the paid layer is node isolation, fleet management, and everything a customer-facing cloud needs around it. Where that line falls, feature by feature, is Section 3.

2

Why it's tempting for a GPU cloud

Selling Kubernetes access to customers normally forces a bad choice: shared namespaces (weak — no CRDs, no cluster-admin, one noisy API server) or a dedicated cluster per customer (strong but slow and expensive). Virtual clusters sit in between:

Customer wants cluster-adminThey get a real API server: own CRDs, operators, webhooks, any K8s version — without touching yours.
Provision in secondsA tenant cluster is one pod + a namespace. No VM boot, no bare-metal churn between customers.
Density pays the billsHundreds of control planes per host cluster; idle tenants cost almost nothing (and can sleep — paid).
Blast-radius containmentA tenant crashing their API server or flooding etcd hurts only their vcluster, not your host control plane.
GPUs stay pooledShared GPU nodes serve many tenants; vcluster passes through nvidia.com/gpu requests untouched.
Certified KubernetesTenant clusters pass CNCF conformance (and the new AI conformance) — customers keep portability.
The core promise: the density of namespaces with the API of dedicated clusters. The catch for a GPU cloud: the isolation that makes it safe to sell, and the platform that makes it operable, are mostly not in the OSS core.

What the Apache 2.0 core actually includes

  • CONTROL PLANEPer-tenant API server + controller manager (vanilla upstream K8s; the k3s distro was removed in v0.33).
  • SYNCERBi-directional object sync for core resources: pods, services, endpoints, ConfigMaps, Secrets, PVCs, ingresses, NetworkPolicies.
  • STORESQLite/kine default, or external etcd you operate. (Embedded managed etcd = gated.)
  • SCHEDULERVirtual scheduler inside the tenant cluster (controlPlane.distro.k8s.scheduler.enabled) with node syncing — OSS since v0.26.
  • TENANCY MODESShared nodes and label-pinned dedicated node pools. (Tenant-joined Private Nodes = gated.)
  • POLICYPod Security Standards enforcement, ResourceQuota / LimitRange, NetworkPolicy creation per tenant.
  • SNAPSHOTSManual vcluster snapshot / restore CLI. (Scheduled/auto snapshots = gated.)
  • EXTRASPlugin SDK, Cluster API provider, Terraform provider, vind (vCluster-in-Docker), metrics-server integration.

The line — what's OSS, what's gated, what's missing

One picture: a host cluster running two tenant vclusters on shared GPU nodes. Green is Apache 2.0 — yours forever. Copper is license-gated — free tier below 32 GPUs, Enterprise contract above. Blue is simply absent from the product: the parts a GPU cloud has to build (or buy elsewhere) no matter what.

Apache 2.0 — free forever license-gated (Free ≤32 GPU / Enterprise) absent — build or buy elsewhere
FIG. 3-1 — The OSS engine (green) gives every tenant a control plane on your shared nodes. Nearly everything that hardens or operationalizes it (copper) is gated; the customer-facing cloud layer (blue) doesn't exist in any tier.
Read the copper column as a price list. vCluster Labs' own Free-tier announcement says the quiet part out loud: features stay closed because otherwise "many enterprises would build around the open source project instead of purchasing" — and it would "enable competitors." A GPU cloud is precisely the persona that boundary is designed to charge.

The license boundary, verified in source

This isn't marketing-page archaeology — the gate is a function. config/config.go in the repo (cloned today, commit 717d0ee1) decides whether your vcluster.yaml trips the pro check, and the Helm chart ships the commercial image by default.

// config/config.go — abridged, real code
func (c *Config) IsProFeatureEnabled() bool {
  if c.ControlPlane.BackingStore.Etcd.
       Embedded.Enabled            { return true }
  if c.Sync.ToHost.Pods.
       HybridScheduling.Enabled    { return true }
  if c.PrivateNodes.Enabled &&
     !c.ControlPlane.Standalone.
       Enabled                     { return true }
  if c.Sync.ToHost.ResourceClaims.
       Enabled                     { return true }
  if c.Sync.FromHost.DeviceClasses.
       Enabled                     { return true }
  // + external DB, CRD sync, ns sync,
  //   embedded CoreDNS, resolveDNS,
  //   central admission, hostPathMapper…
}
Note what's on the list for a GPU cloud: DRA sync (ResourceClaims / DeviceClasses) and hybrid scheduling are pro-gated in code. The modern GPU-allocation path is behind the license.

# vcluster.yaml — pin the Apache 2.0 build
controlPlane:
  statefulSet:

      repository: loft-sh/vcluster-oss
  distro:
    k8s:
      scheduler:
        enabled: true   # virtual sched = OSS
sync:
  fromHost:
    nodes:
      enabled: true   # real node info

Alternatives — the honest comparison

Every credible way to sell "your own Kubernetes" on shared GPU infrastructure, compared. Note what the top-end neoclouds actually run: CoreWeave hands each customer dedicated bare metal behind BlueField-3 DPUs; Nebius wraps GPU nodes in KubeVirt VMs; Crusoe uses cloud-hypervisor VMs. Nobody at the top sells shared-kernel tenancy.

  • PICK vCLUSTER OSS WHEN tenants are internal or trusted, density is the point, and you accept shared-kernel risk with PSS + quotas as the only fence.
  • PICK vCLUSTER ENTERPRISE WHEN you want the packaged GPU-cloud platform (Private Nodes + Auto Nodes + BCM + UI) and would rather pay per node than staff a platform team for 6–12 months.
  • PICK KAMAJI + CAPI WHEN you're committed to building: hard isolation, zero license risk, every component Apache 2.0 — and you own the roadmap instead of renting it.
  • PICK PER-TENANT METAL/VMs WHEN customers are large, security-sensitive, and buying whole nodes anyway — multi-node training tenants usually are.
The realistic hybrid for a GPU cloud: hosted control planes (vcluster OSS or Kamaji) + dedicated GPU node pools per paying tenant. Whole-node tenancy sidesteps the shared-kernel problem, GPU customers rarely buy fractions of an 8-GPU box anyway, and the control-plane tech becomes an implementation detail rather than a security boundary.

Where it falls short — the build-it-yourself ledger

Concretely: run vcluster OSS and nothing else, and here is every gap between "tenant control planes work" and "customers pay you money" — with the paid shortcut and the OSS path for each.

OSS PATH EXISTS

Kernel-level isolation

Shared nodes = one kernel for all tenants; a container escape owns every tenant on the box. The paid answer is vNode; the OSS answer would be Kata VM-isolated pods with GPU/MIG passthrough — real work, but proven.

kata containers · gvisor (weak GPU) · vnode (paid)

MUST BUILD

Tenant network isolation

All tenants share the host CNI; NetworkPolicy is the only fence and only works if the CNI enforces it. No per-tenant VPC/VLAN in any free tier — that's the Netris partnership (Enterprise). Building VLAN/EVPN or IB partition (pkey) isolation in the fabric is on you.

cilium netpols · EVPN/VXLAN · IB pkeys · netris (commercial)

OSS PATH EXISTS

GPU scheduling policy

Gang scheduling, bin-packing, fractional GPUs, MIG carving — none of it ships in OSS vcluster, and hybrid scheduling (host-level custom schedulers for tenant pods) is Enterprise-gated. NVIDIA's KAI Scheduler is Apache 2.0; wiring it per-tenant is on you.

kai-scheduler · gpu-operator · MIG / time-slicing

MUST BUILD

Metering & billing

Nothing in any tier meters tenants for invoicing. vBilling (their new OSS side-project) is alpha: GPU-hours by SKU via DCGM, streaming to Lago. Usable as a starting point, but revenue-grade metering is your code either way.

vbilling (alpha) · openmeter · lago · dcgm-exporter

GATED — OR BUILD

Tenant lifecycle & self-service

Signup → provision → kubeconfig delivery → quotas → SSO → sleep idle tenants → delete. That's the entire vCluster Platform value prop (Free ≤32 GPUs, then Enterprise). DIY = a control plane of your own: operator + portal + templates.

platform (paid) · custom operator + CAPI provider · backstage

GATED — OR BUILD

HA state & node lifecycle

OSS backing store is SQLite or self-run etcd — embedded managed etcd is gated, and HA beyond one vcluster is Free-tier-capped. Auto Nodes (Karpenter provisioning, BCM/Metal3 providers) is gated too; the OSS route is Cluster API + Karpenter glue you own.

external etcd ops · cluster api · karpenter

Ledger total: two gaps have solid OSS answers you integrate (Kata, KAI), two are yours to build under any vendor (billing, fabric isolation), and two are the actual buy-vs-build decision (Platform layer, node lifecycle). That last pair is what an Enterprise contract — anchored around £2,000–4,000 per node per year on 2024 UK G-Cloud pricing — is really priced against.

Gotchas — before anyone writes a line of code

  • Shared nodes are soft tenancy, full stop. Pod Security Standards, quotas, and NetworkPolicies are policy, not a boundary. If two competing customers' training jobs share a kernel and an NVLink fabric, a single container escape or noisy neighbor is a company-ending incident, not a bug. Sell whole nodes per tenant or add Kata/vNode before selling to strangers.
  • The default Helm install is the commercial image. chart/values.yaml points at loft-sh/vcluster-pro; with gated config and no license it crashloops. Pin ghcr.io/loft-sh/vcluster-oss deliberately, everywhere, or a teammate's "harmless" config flag becomes a licensing dependency.
  • The Free tier is a demo, not a tier. 64 vCPUs or 32 GPUs total and one HA vcluster — four H100 boxes — with license validation phoning admin.loft.sh. Losing connectivity degrades gracefully to OSS features, but building on Free-tier features means building on a license you don't control.
  • The GPU-native path is gated. DRA sync and hybrid scheduling (host GPU schedulers for tenant pods) trip IsProFeatureEnabled(). The OSS workaround — provider-owned GPU stack + extended resources + in-tenant virtual scheduler — works today but is the legacy allocation model, and the industry is moving to DRA.
  • NetworkPolicy isolation is only as real as the host CNI. vcluster happily syncs policies to a CNI that ignores them. Cilium/Calico with enforcement verified per-cluster is table stakes before tenant #2.
  • History says the line moves. k3s/k0s/EKS distros: removed. v0.20: pro image became the default. Jan 2026: Enterprise features moved to a capped Free tier, explicitly to stop people "building around the open source project." Architect so that gated features are swappable, not load-bearing.
  • The syncer is host-cluster attack surface. Each tenant's syncer holds real RBAC in a host namespace, and every tenant API server is a new internet-facing door to your infrastructure. The hardening guides are good — and entirely on you to apply. (Isolated Control Plane, which separates CP and workload clusters, is Enterprise.)
  • You can verify all of this in an afternoon. Section 9 reproduces the license gate, the OSS install, and the Kamaji comparison on a laptop — before any architecture doc cites this sheet.

Run it yourself — verify the claims

Three labs, escalating from laptop to decision-grade evidence. LAB A reproduces this sheet's central claim — the license gate — with your own eyes.

LAB A — Touch the OSS line on your laptop

~15 min · needs Docker + kind + helm

  1. Create a throwaway host cluster and install the CLI.

    brew install kind loft-sh/tap/vcluster   # or: curl -fsSL https://kind.sigs.k8s.io
kind create cluster --name host

    EXPECT: `kubectl get nodes` shows one kind node — this plays the role of your GPU fleet.

  2. Create a tenant on the true OSS image (not the chart default):

    cat <<'EOF' > oss.yaml
controlPlane:
  statefulSet:
    image: { repository: loft-sh/vcluster-oss }
EOF
vcluster create tenant-a -n tenant-a -f oss.yaml

    EXPECT: ~60s later your kubecontext is inside tenant-a. `kubectl get ns` shows a fresh cluster; the host only sees a pod.

  3. Prove the isolation model. Create a CRD in the tenant, then look for it on the host:

    kubectl create ns demo && kubectl get crds | head
vcluster disconnect
kubectl get ns | grep demo   # nothing — tenant objects are virtual

    EXPECT: tenant namespaces/CRDs don't exist on the host. Only synced pods appear, renamed, in namespace tenant-a.

  4. Now trip the gate. Add one copper-zone flag from Section 4 and upgrade:

    cat <<'EOF' > gated.yaml
controlPlane:
  statefulSet:
    image: { repository: loft-sh/vcluster-oss }
  backingStore: { etcd: { embedded: { enabled: true } } }
EOF
vcluster create tenant-a -n tenant-a --upgrade -f gated.yaml

    EXPECT: rejection/crashloop — embedded etcd is a pro feature and the OSS image doesn't even contain it. You just met IsProFeatureEnabled().

LAB B — The Kamaji comparison

~45 min · same kind cluster

  1. Install Kamaji on the same kind cluster (needs cert-manager first):

    helm repo add jetstack https://charts.jetstack.io && helm repo add clastix https://clastix.github.io/charts
helm install cert-manager jetstack/cert-manager -n cert-manager --create-namespace --set crds.enabled=true
helm install kamaji clastix/kamaji -n kamaji-system --create-namespace

    EXPECT: kamaji controller Running. Follow the getting-started guide for a TenantControlPlane CR.

  2. Create a TenantControlPlane and note the difference. Kamaji gives you an API server the same way vcluster does — but no workers and no syncer. Workers are real machines you join with a bootstrap token (in prod: Cluster API).

    kubectl apply -f https://raw.githubusercontent.com/clastix/kamaji/master/config/samples/kamaji_v1alpha1_tenantcontrolplane.yaml
kubectl get tcp -A

    EXPECT: a READY tenant control plane pod. The mental model shift: vcluster fakes nodes via the syncer; Kamaji expects you to bring real ones — which is exactly why its isolation is hard and its platform burden is bigger.

  3. Write down the comparison while it's fresh. Provisioning feel, what tenants can see, what you'd still have to build for node lifecycle in each. That half-page is the core of any architecture decision.

LAB C — Graduate to one real GPU node

optional · 1 spare GPU box · the decision-maker

  1. Stand up a one-node host cluster on a spare GPU machine (k3s or kubeadm), install the GPU Operator on the host, and confirm nvidia.com/gpu is allocatable.

    helm install --wait gpu-operator nvidia/gpu-operator -n gpu-operator --create-namespace
kubectl describe node | grep -A3 'nvidia.com/gpu'

    EXPECT: the node advertises its GPUs. This is the provider-owned GPU stack from FIG. 3-1.

  2. Create an OSS tenant and run a GPU pod from inside it.

    vcluster create gpu-tenant -n gpu-tenant -f oss.yaml
kubectl run cuda --image=nvcr.io/nvidia/cuda:12.4.1-base-ubuntu22.04 \
  --overrides='{"spec":{"containers":[{"name":"cuda","image":"nvcr.io/nvidia/cuda:12.4.1-base-ubuntu22.04","command":["nvidia-smi"],"resources":{"limits":{"nvidia.com/gpu":1}}}]}}' --restart=Never
kubectl logs cuda

    EXPECT: nvidia-smi output from inside the tenant — GPU passthrough works on pure OSS. Then note what didn't happen: no per-tenant GPU quota, no gang scheduling, no metering record, no network isolation. That gap list is Section 6, live.

  3. Time-box a DCGM metering spike. Point dcgm-exporter + Prometheus at the node and attribute GPU-seconds to the tenant namespace label. If that takes under a day, the billing gap is buildable; if not, that's real data for the buy side of the ledger.

Glossary — the jargon, decoded

Syncer
vCluster's translation engine: copies pods/services/PVCs from the tenant's virtual API server into a host namespace (renamed), and node/event info back up. It holds real host RBAC — which makes it both the magic and the attack surface.
kine / SQLite backing store
An etcd-API shim over SQL databases. OSS vcluster stores tenant state in SQLite via kine by default; managed embedded etcd is license-gated. github.com/k3s-io/kine
Private Nodes
License-gated mode where real worker machines join the tenant's virtual control plane directly — own kubelet, CNI, and GPU Operator per tenant. vCluster's own docs call this the model for GPU clouds.
Hybrid scheduling
Enterprise feature letting tenant pods be placed by host-level custom schedulers (e.g. a shared GPU scheduler like KAI) instead of the default host scheduler.
DRA — Dynamic Resource Allocation
Kubernetes' modern device-allocation API (ResourceClaims/DeviceClasses) replacing bare extended resources like nvidia.com/gpu. vCluster can sync DRA objects — but only behind the license gate. k8s docs
Sleep mode
Platform feature that scales idle tenant workloads to zero and wakes them on the next request — the density multiplier for internal dev tenants. Enterprise-gated.
MIG / time-slicing
Two ways to split a physical GPU: MIG carves hardware partitions (A100/H100); time-slicing shares whole GPUs in software with no memory isolation. Both configured in the node's GPU stack, below any tenancy layer.
Cluster API (CAPI)
Kubernetes-native machinery for creating and managing clusters and machines declaratively. The glue for any DIY hosted-control-plane platform (Kamaji, k0smotron both ship CAPI providers). cluster-api.sigs.k8s.io
Hosted control plane
Running a tenant's API server/etcd as pods in a management cluster instead of on dedicated masters. The pattern behind vcluster, Kamaji, k0smotron, HyperShift, and GKE/EKS internals alike.
Kata Containers
OCI runtime that wraps each pod in a lightweight VM — kernel-level isolation on shared nodes. GPU use = VFIO passthrough of a whole GPU or MIG slice. The OSS counterpart to vCluster's paid vNode. katacontainers.io
Karpenter
Node-autoscaling controller that provisions right-sized machines on demand. vCluster's "Auto Nodes" is gated Karpenter integration; using Karpenter directly is Apache 2.0. karpenter.sh
DPU / BlueField
A NIC with its own CPU that enforces network/storage isolation below the host OS. CoreWeave's tenancy boundary — and the part of the top-end pattern with no OSS equivalent.
Pod Security Standards (PSS)
Kubernetes' built-in pod restriction profiles (privileged/baseline/restricted). OSS vcluster can enforce "baseline" so privileged/hostPath pods never sync to the host — policy-level protection only.
Open-core
Business model: a genuinely open engine plus proprietary features and a management plane around it. The strategic risk isn't today's feature split — it's that the vendor controls where the line moves next.
IB pkeys
InfiniBand partition keys — VLAN-like tenant separation on the training fabric. Multi-tenant GPU clouds need this at the fabric layer; no Kubernetes tenancy product provides it.
DCGM
NVIDIA's Data Center GPU Manager — the telemetry source (utilization, memory, power) for GPU metering. dcgm-exporter feeds it to Prometheus; vBilling and every billing pipeline sit on top. dcgm-exporter
