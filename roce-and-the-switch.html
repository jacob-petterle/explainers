RoCE & THE SWITCH

How GPUs read each other's memory straight over Ethernet — and what the switch in the middle is actually doing (spoiler: it is not "transparent").

1

What they are

HardwareU-SW1

A phonebook with wires: the Ethernet switch.

A switch is a box of ports plus one crucial data structure: a MAC address table ("who lives on which port"). It builds that table by watching: every frame that arrives teaches it "MAC X lives behind port N."

When a frame arrives for a MAC it knows, the switch sends it out one port. When it arrives for a MAC it doesn't know — or for the broadcast address, like every ARP request — the switch floods it out every port in the same broadcast domain. That flooding is why a switch is not "transparent glass": by default, everyone plugged into the same L2 domain receives each other's broadcasts and can lie about who they are. VLANs carve one physical switch into separate broadcast domains, which is the basic tenant-isolation primitive.

ProtocolIBTA ANNEX A17

InfiniBand's memory language in an Ethernet envelope: RoCE.

RoCE (pronounced "rocky") is RDMA — Remote Direct Memory Access — carried over ordinary Ethernet. RDMA lets one machine's NIC write into another machine's RAM (or GPU memory) directly, with no CPU, no kernel, and no copies on the data path.

The modern version, RoCEv2, wraps InfiniBand's transport headers (IBTH/RETH) inside a routable UDP/IP packet on destination port 4791. It's the standard defined by the InfiniBand Trade Association, and it's how most Ethernet-based GPU clusters move gradients, KV-caches, and model weights today.

The relationship in one sentence: RoCE is a guest with extreme demands, and the switch fabric is the house that must be renovated to host it — RDMA assumes packets basically never drop, so the switches must be configured to be (nearly) lossless, and because RDMA carries no real authentication, the switches must also provide the tenant isolation.

2

The job it solves: moving memory, not messages

Normal networking (TCP/IP) treats data as a stream of messages: the sender's CPU copies data into the kernel, the kernel makes packets, the receiver's kernel reassembles them, and the receiver's CPU copies them out to the application. At GPU-cluster speeds that CPU-and-copies tax is fatal. RDMA's answer: the NICs talk directly to each other's memory.

The core promise: the network disappears. A remote GPU's memory becomes, from the application's point of view, just slow(ish) local memory — which is the entire premise of training or serving one model across many machines.

What RDMA actually gives you

  • ZERO-COPYData goes NIC-to-application-memory. No intermediate buffers, no memcpy chain.
  • KERNEL BYPASSApps post work to the NIC via queue pairs from user space; the OS is not on the data path.
  • CPU BYPASSOne-sided READ/WRITE ops complete without the remote CPU even being interrupted.
  • IN SILICONSegmentation, reassembly, ordering, and retransmission run on the NIC, not in software.
  • THE CATCHAll of this assumes a cooperative fabric: near-zero loss, and trust decided elsewhere. Sections 3–4.
3

What a switch actually does — one frame, two fates

Follow two frames through the same leaf switch. The switch never looks at IP addresses — it matches destination MAC against its learned table, scoped by VLAN. A known destination exits one port. An unknown destination, or a broadcast (like every ARP request), is copied to every port in that VLAN — and stops dead at the VLAN boundary. That boundary is tenant isolation.

known unicast — one port broadcast / unknown — flooded in-VLAN tenant A domain tenant B domain
FIG. 3-1 — The switch's whole job: learn source MACs, forward known unicast to one port, flood everything else to the VLAN. The VLAN boundary is where L2 tenant isolation happens.
Why this matters for RoCE: a RoCEv2 packet is just a UDP packet — the switch forwards it by MAC and VLAN like anything else. So (1) two tenants in one VLAN can reach each other's RDMA NICs, and RoCE itself will barely object (see Gotcha 01); and (2) the same switch must be configured with priority flow control and ECN so those RDMA flows almost never experience a dropped packet. The switch is doing two extra jobs for RoCE: bouncer and shock absorber.

Hands-on mechanics — what's actually on the wire

Four views of the same machinery: the packet itself, the switch's brain, the "lossless" knobs, and the isolation config. Everything here is reproducible in Section 9's labs.

## Anatomy of one RoCEv2 packet (outside → in)

ETHERNET   dst MAC · src MAC          ← all the switch reads for forwarding
802.1Q     VLAN tag, e.g. vid 100     ← the tenant label
IPv4/v6    src/dst IP · ECN bits      ← congestion is signaled here (2 bits)
UDP        dst port 4791              ← "this is RoCEv2"
IBTH       opcode · QP number · PSN   ← InfiniBand transport header
RETH       remote VA + RKEY           ← WHERE in the remote machine's memory
PAYLOAD    up to 4096 B of your data
ICRC/FCS   integrity checksums
Takeaway: the "address" that matters to RDMA is a raw memory address plus an RKEY capability token — not a URL, not a socket. Notice what's missing: any user identity, session, or encryption. Trust lives entirely in the fabric config (VLAN tag) and the guessability of QP/RKEY values.
# The switch's brain, on any Linux box acting as a bridge:
$ bridge fdb show br br0
aa:c1:ab:11:22:01 dev vethA vlan 100 master br0        ← learned: GPU1 → that port
aa:c1:ab:11:22:02 dev vethB vlan 100 master br0
bb:c1:ab:33:44:03 dev vethC vlan 200 master br0

# Same idea on a real NVIDIA/Cumulus switch:
$ nv show bridge domain br_default mac-table
Takeaway: forwarding state is learned, per-VLAN, and finite. Entries age out (default ~5 min on Linux); the table has a hardware size limit — which is exactly what a MAC-flooding attack exhausts to force the switch into flood-everything mode (Gotcha 02).
# RDMA hates drops, so the fabric gets two mechanisms:
#  PFC (IEEE 802.1Qbb): receiver says "PAUSE priority 3" hop-by-hop — a handbrake
#  ECN + DCQCN: switch marks packets when queues grow; NICs slow down — a dimmer

# On NVIDIA Cumulus Linux 5.x this entire tuning ritual is one command:
$ nv set qos roce                    # lossless mode: PFC on prio 3 + ECN
$ nv config apply
$ nv show qos roce                   # verify: pfc-priority 3, ecn enabled

# On the host NIC (Mellanox/NVIDIA ConnectX):
$ mlnx_qos -i eth0 --pfc 0,0,0,1,0,0,0,0   # pause-enable priority 3 only
Takeaway: "lossless Ethernet" is not a property Ethernet has — it's a fragile agreement you configure, per priority class, on every switch and NIC in the path. PFC stops loss but can cause head-of-line blocking and pause storms; ECN/DCQCN prevents the congestion in the first place. Production fabrics use both.
# Tenant isolation, smallest to largest:

# 1. VLAN per tenant on the switch port (single switch / small pod)
$ nv set interface swp1-swp2 bridge domain br_default access 100  # tenant A
$ nv set interface swp3-swp4 bridge domain br_default access 200  # tenant B

# 2. VXLAN/EVPN (many switches, many buildings): same idea, 24-bit IDs,
#    tunneled over IP, with BGP distributing the MAC tables between switches

# 3. DPU-enforced ACLs (BlueField): rules run on the NIC's own Arm cores —
#    the tenant has root on the host and still can't touch them
Takeaway: these are the same primitive at three scales — a tag that scopes the broadcast domain and forwarding table. Products like Netris or OVN are control planes that automate pushing exactly this config; the data plane underneath is identical.

  • PICK ROCEEthernet-based GPU fabric on commodity switches; you own the tuning and supply isolation via VLAN/VXLAN. Today's default for Ethernet neoclouds (NVIDIA Spectrum-X is a tuned RoCE system).

Where you'll meet it

Training collectives

NCCL's all-reduce/all-gather operations — the gradient sync of every distributed training step — run on RDMA verbs over RoCE or IB, with GPUDirect so data never touches host RAM.

NCCL · nccl-tests · RCCL (AMD)

Disaggregated inference

Splitting prefill and decode across nodes means shipping KV-caches between GPUs mid-request. RDMA is what makes that cheap enough to be worth doing.

NIXL · NVIDIA Dynamo · vLLM

Storage fabrics

NVMe-over-Fabrics and parallel filesystems use RDMA to serve model weights and checkpoints at near-local speed — often sharing the same lossless Ethernet fabric on a different priority class.

NVMe-oF · WEKA · VAST · Ceph (RDMA messenger)

Multi-tenant GPU clouds

Renting GPU clusters to strangers means strangers share your fabric. VLAN/VXLAN-per-tenant plus DPU-enforced ACLs is the standard answer — and an explicit audit item for GPU-cloud security ratings.

Gotchas — security first

  1. RoCE has no login prompt. RoCEv2 carries no authentication and no encryption — access control is a 24-bit QP number and an RKEY, both of which research (ReDMArk, USENIX Security '21) showed are often predictable. If a hostile machine can send UDP/4791 to your NIC, you must assume it can eventually read or corrupt memory. Fabric isolation (VLAN/VXLAN per tenant, DPU ACLs) isn't hardening — it's the only lock on the door.
  2. Switches flood, and tenants can lie. Within one VLAN, every host sees every ARP broadcast, can answer "that IP is me" (ARP spoofing), and can blast bogus source MACs until the switch's table overflows and it floods all traffic to everyone (MAC flooding). A tenant with root on a bare-metal box can do all of this. An IP firewall never sees it, because it happens below IP. One VLAN per tenant, no exceptions.
  3. PFC is a handbrake that can jam. Per-priority pause stops loss, but pause frames propagate hop-by-hop: one slow receiver or one miscabled port can stall an entire priority class fabric-wide (pause storms, head-of-line blocking, even deadlock). This is why real deployments lean on ECN/DCQCN to prevent congestion and keep PFC as the last resort.
  4. The tuning is per-hop and unforgiving. PFC priority, ECN thresholds, and MTU must agree on every NIC and every switch in the path. One mismatched box silently gives you drops, stalls, or half the bandwidth — and NCCL will just look "mysteriously slow." Always benchmark the fabric (nccl-tests, ib_send_bw) before blaming the model code.
  5. "Converged" is aspirational. The C in RoCE promised one Ethernet for everything. In practice AI fabrics dedicate a backend network to RDMA (or at least a strict priority class), separate from tenant north-south and management traffic. Budget switch ports accordingly.
  6. Watch the successor. RoCE's flaws (single-path flows, loss fragility, no security) are exactly what Ultra Ethernet fixes. It won't obsolete your fabric tomorrow, but every 2026+ NIC/switch purchase should be checked for UEC readiness.
  7. Don't take this sheet's word for it. Every claim above — flooding, MAC learning, real RDMA packets — is reproducible on your laptop in the labs below.

Run it yourself

LAB A — Watch a switch think

~10 min · any Linux (VM/WSL2 fine) · zero installs

Linux can be a switch. You'll build one, plug in two fake hosts (network namespaces), and watch it learn MACs and flood ARP — FIG. 3-1, reproduced by you.

  1. Build the switch and two hosts.

    sudo ip link add br0 type bridge && sudo ip link set br0 up
for h in A B; do sudo ip netns add $h; done
sudo ip link add vA type veth peer name pA; sudo ip link add vB type veth peer name pB
sudo ip link set vA netns A; sudo ip link set vB netns B
sudo ip link set pA master br0 up; sudo ip link set pB master br0 up
sudo ip netns exec A ip addr add 10.9.0.1/24 dev vA; sudo ip netns exec A ip link set vA up
sudo ip netns exec B ip addr add 10.9.0.2/24 dev vB; sudo ip netns exec B ip link set vB up

    EXPECT: no output — a bridge (switch) br0 with "hosts" A and B plugged into ports pA and pB.

  2. Look at the empty phonebook, then make it learn.

    bridge fdb show br br0 | grep -v permanent      # (nearly) empty
sudo ip netns exec A ping -c2 10.9.0.2
bridge fdb show br br0 | grep -v permanent      # now populated

    EXPECT: after the ping, two learned entries — one MAC on pA, one on pB. That table IS the switch.

  3. Prove the flood. Watch host B's wire while A asks for an IP that doesn't exist.

    sudo ip netns exec B tcpdump -i vB -n arp &     # eavesdrop as host B
sudo ip netns exec A ping -c2 -W1 10.9.0.99    # nobody has this IP

    EXPECT: B's tcpdump prints A's "who-has 10.9.0.99" ARP requests — traffic that was never addressed to B. The switch flooded it. This is why "a firewall on your IP" doesn't isolate tenants.

  4. Now add the VLAN wall and watch the same flood stop.

    sudo ip link set br0 type bridge vlan_filtering 1
sudo bridge vlan add dev pA vid 100 pvid untagged; sudo bridge vlan del dev pA vid 1
sudo bridge vlan add dev pB vid 200 pvid untagged; sudo bridge vlan del dev pB vid 1
sudo ip netns exec A ping -c2 -W1 10.9.0.99    # tcpdump on B stays silent

    EXPECT: nothing arrives at B anymore. Ports in different VLANs = different broadcast domains = tenant isolation. Clean up: sudo ip netns del A; sudo ip netns del B; sudo ip link del br0

LAB B — Real RDMA, no RDMA hardware

~15 min · Linux with kernel 4.8+ · apt install

Soft-RoCE implements RoCEv2 in the kernel on any ordinary NIC. You'll run genuine ibverbs traffic — same API and packets as a $2,000 ConnectX — on your laptop.

  1. Install the RDMA userspace stack and load the driver.

    sudo apt install -y rdma-core ibverbs-utils rdmacm-utils perftest
sudo modprobe rdma_rxe
sudo rdma link add rxe0 type rxe netdev eth0   # use your interface name (ip addr)

    EXPECT: `ibv_devices` now lists a device named rxe0.

  2. RDMA ping yourself. Two terminals:

    rping -s -a 127.0.0.1 -v &                     # terminal 1: server
rping -c -a 127.0.0.1 -v -C 4                  # terminal 2: client, 4 pings

    EXPECT: four "ping data: rdma-ping-N" lines. Those round trips used RDMA READ/WRITE verbs, not sockets.

  3. Measure RDMA bandwidth with the industry-standard tool:

    ib_send_bw -d rxe0 &                           # server
ib_send_bw -d rxe0 127.0.0.1                   # client

    EXPECT: a bandwidth report table (BW average, MsgRate). On Soft-RoCE the number is modest — the point is the mechanics, and that this exact command is how real GPU fabrics get acceptance-tested.

  4. See the packets from Section 4 with your own eyes.

    sudo tcpdump -i lo -n udp port 4791 -c 6 &
rping -c -a 127.0.0.1 -v -C 2

    EXPECT: UDP packets to port 4791 — RoCEv2 on the wire, exactly the envelope from the PACKET tab.

LAB C — Graduation: a real AI-switch fabric, free

~45 min · free account · browser only

  1. Spin up virtual Spectrum switches. NVIDIA Air is a free hosted simulator running the real Cumulus Linux switch OS. Create an account and launch a prebuilt leaf-spine topology.

    EXPECT: SSH access to a leaf switch running the same NOS as production GPU fabrics.

  2. Do the operator moves from Section 4: put two ports in different VLANs (nv set interface swp1 bridge domain br_default access 100), then enable lossless RoCE (nv set qos roce, nv config apply) and inspect it (nv show qos roce).

Glossary — the jargon, decoded

MAC address
The burned-in Layer-2 hardware address of a NIC (e.g. aa:c1:ab:11:22:01). Switches forward on MACs; routers forward on IPs. wikipedia
Broadcast domain
The set of ports that receive each other's broadcast frames. One un-VLANed switch = one big broadcast domain = zero tenant isolation.
Flooding
What a switch does with a frame whose destination it hasn't learned (or a broadcast): copy it out every port in the VLAN. Normal, constant, and the reason L2 neighbors always see some of each other's traffic.
ARP
Address Resolution Protocol — "who has IP 10.0.0.5? tell me your MAC," sent as a broadcast. Trivially spoofable by anyone in the same VLAN, which is the classic L2 attack. wikipedia
VLAN (802.1Q)
A 12-bit tag in the Ethernet header that splits one physical switch into up to ~4094 separate broadcast domains. The basic per-tenant wall. wikipedia
VXLAN
VLANs, scaled: a 24-bit tenant ID (~16M) carried in a UDP tunnel, so a tenant's L2 domain can span many switches and sites over routed IP. wikipedia
EVPN
The control plane for VXLAN at scale: BGP distributes "which MAC lives behind which switch" so flooding is minimized across the fabric (RFC 8365). rfc 8365
RDMA
Remote Direct Memory Access: NIC-to-NIC reads/writes of application (or GPU) memory with no CPU or kernel on the data path. Delivered by InfiniBand, RoCE, iWARP, or UET. wikipedia
Queue pair (QP)
RDMA's "connection": a send queue + receive queue an application posts work requests to, identified by a 24-bit number. The verbs API manipulates QPs.
IBTH / RETH · RKEY
The InfiniBand transport headers inside every RoCE packet. RETH carries the remote virtual address plus the RKEY — the capability token that authorizes touching that memory region.
PFC (802.1Qbb)
Priority Flow Control: a receiver tells the previous hop "pause priority N" so its buffer never overflows. The blunt instrument that makes Ethernet lossless — with head-of-line-blocking side effects.
ECN
Explicit Congestion Notification: 2 bits in the IP header a switch sets when queues grow, telling endpoints "slow down before I drop." wikipedia
DCQCN
The congestion-control algorithm RoCE NICs run in response to ECN marks (Microsoft/Mellanox, SIGCOMM 2015). The "dimmer switch" that keeps PFC from ever firing. paper
Lossless Ethernet
Ethernet configured (PFC + ECN + buffer tuning, per priority class) so packets are ~never dropped. A fragile achievement, not a default.
GPUDirect RDMA
NVIDIA tech letting the RDMA NIC DMA directly into/out of GPU memory, skipping host RAM entirely. nvidia docs
PKey
InfiniBand's in-protocol tenant partition key — membership checked on every packet by the fabric. The thing RoCE doesn't have, which is why RoCE isolation must come from VLANs/ACLs instead.
DPU
Data Processing Unit (e.g. NVIDIA BlueField): a NIC with its own Arm cores running the operator's network/ACL policy — an enforcement point the tenant can't reach even with root on the host.
Soft-RoCE (rxe)
The Linux kernel's software implementation of RoCEv2 (rdma_rxe module) — real verbs API and real RoCE packets on any Ethernet NIC. Slow, but perfect for learning and CI. rdma-core
