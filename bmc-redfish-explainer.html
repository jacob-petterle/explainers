1

What they are

HardwareU-BMC1

A server inside your server.

The BMC is a small, dedicated system-on-chip soldered onto the motherboard (a common example is the ASPEED AST2600, an ARM SoC). It runs its own firmware — often a small embedded Linux — on its own processor, with its own memory and its own network port.

Crucially, it lives on the standby power rail: as long as the power supply has AC, the BMC is alive — even when the server itself is "off," crashed, or has no OS installed at all.

Software / ProtocolDSP0266

A REST API for hardware.

Redfish is the open standard, maintained by the DMTF, that defines how you talk to a BMC: an HTTPS API returning JSON, organized as a browsable tree of resources under /redfish/v1/. The spec itself is free to download.

Instead of vendor tools and cryptic binary protocols, managing a physical server becomes indistinguishable from calling any modern web API — curl, Python, or Ansible all work out of the box.

The relationship in one sentence: the BMC is the agent, Redfish is the language. The BMC has physical hooks into everything on the board; Redfish is the standardized way to pull those hooks over the network.

2

The job it solves: out-of-band management

Normal remote administration — SSH, RDP, agents — is in-band: it rides on the server's own OS and NIC. That works only while everything is healthy. The BMC provides an out-of-band path that works when nothing else does:

OS is frozenSSH is dead. The BMC can hard-reset the box or show you the crash screen.
Server is powered offThe BMC is still on standby power. One API call turns the machine on.
No OS installed yetMount an ISO over the network as "virtual media" and install remotely.
BIOS needs an updatePush firmware through the BMC — no boot USB, no data-center visit.
Is it overheating?Read fan speeds, temperatures, voltages, and power draw from board sensors.
Rack is 3,000 miles awayFull keyboard-video-mouse console over IP, as if you were standing there.
The core promise: "lights-out management." A data center with zero staff on site can still power-cycle, reinstall, diagnose, and update every machine — because each one carries its own independent little administrator.

What a BMC can actually do

  • POWERPower on / off / hard reset / NMI the host, and meter its power draw.
  • SENSORSMonitor temperatures, fan tachometers, voltages, PSU health, intrusion switches.
  • CONSOLERemote KVM-over-IP and Serial-over-LAN — see the screen before the OS even boots.
  • V-MEDIAVirtual media: present a remote ISO to the host as if it were a locally plugged USB drive.
  • FIRMWAREUpdate BIOS/UEFI, the BMC's own firmware, and often NICs, drives, and GPUs.
  • INVENTORYReport exact CPUs, DIMMs, drives, PCIe cards, serial numbers, and firmware versions.
  • LOGSKeep a hardware event log (SEL: ECC errors, thermal trips, PSU failures) that survives OS reinstalls.
  • BOOTSet boot order and one-time boot targets (PXE, ISO, disk) for provisioning workflows.

How it works — block diagram

The BMC sits beside the host system, wired into it through low-level buses (see I2C, eSPI, NC-SI in the glossary). The two power domains are the key to the whole trick: the host domain can be dead while the standby domain keeps the BMC serving requests.

Talking Redfish

Everything the BMC knows is exposed as a tree of JSON resources rooted at /redfish/v1/. You browse it like a filesystem and act on it with plain HTTP verbs: GET to read, PATCH to configure, POST to trigger actions. Every resource type is documented with examples in the Redfish Resource & Schema Guide.

/redfish/v1/ ├── Systems/ # logical servers │ └── 1/ │ ├── Processors/ │ ├── Memory/ │ ├── Storage/ │ └── Actions/ ├── Chassis/ # physical: power, │ └── 1/ # thermal, sensors ├── Managers/ # the BMC itself │ └── BMC/ ├── UpdateService/ # firmware ├── EventService/ # push alerts ├── TelemetryService/ # metrics ├── AccountService/ # users, roles └── SessionService/ # auth tokens
FIG. 4-1 — The three pillars: Systems (logical), Chassis (physical), Managers (the BMC). Services hang alongside — see TelemetryService in the glossary.
# What is this machine, and is it on?
GET https://10.0.0.42/redfish/v1/Systems/1

{
  "Model":        "PowerEdge R760",
  "SerialNumber": "7X2K9M3",
  "PowerState":   "On",
  "ProcessorSummary": { "Count": 2 },
  "MemorySummary": {
    "TotalSystemMemoryGiB": 512
  },
  "Status": { "Health": "OK" }
}
One GET replaces walking to the rack. Health, inventory, and power state — machine-readable, vendor-neutral.
# The OS is hung. Force a reboot.
POST …/Systems/1/Actions/ComputerSystem.Reset

{ "ResetType": "ForceRestart" }

# Other ResetTypes:
#  "On", "ForceOff", "GracefulShutdown",
#  "PowerCycle", "Nmi" …
This works even when SSH is dead — the BMC toggles the power lines directly, below the OS.
# Is the chassis running hot?
GET …/Chassis/1/ThermalSubsystem/ThermalMetrics

{
  "TemperatureSummaryCelsius": {
    "Internal": { "Reading": 46.5 },
    "Intake":   { "Reading": 23.0 }
  }
}
Fan speeds, voltages, and per-component power live under Chassis/. Scrape it on a schedule and you have hardware monitoring with zero agents installed.
# Push a BIOS image to the whole fleet
POST …/UpdateService/update-multipart
Content-Type: multipart/form-data

  UpdateParameters + bios_v2.14.bin

# → returns a Task you can poll:
GET …/TaskService/Tasks/545
{ "TaskState": "Running", "PercentComplete": 60 }
Long operations return Tasks — async job handles you poll or subscribe to. Firmware for hundreds of machines becomes a for-loop.

Why Redfish replaced IPMI

For ~20 years the answer here was IPMI (Intelligent Platform Management Interface, 1998). It defined the BMC's classic role but aged badly: a binary protocol over UDP, well-known cryptographic weaknesses, and no standard model for modern hardware. The DMTF shipped Redfish v1.0 in 2015 as the successor; IPMI is now legacy, though it still lingers on many boards.

AspectIPMI (1998)Redfish (2015 → today)
TransportBinary packets over UDP (RMCP+)HTTPS + JSON — firewall- and human-friendly
SecurityWeak by design (e.g. the RAKP handshake leaks password hashes; "cipher 0" allowed no auth at all)TLS, session tokens, role-based accounts, audit-friendly
Data modelNumeric sensor codes, vendor decoding requiredVersioned, self-describing JSON schemas
Toolingipmitool and vendor CLIsAnything that speaks HTTP: curl, Python, Ansible, Terraform
ScopeBasic power, sensors, serial consoleFull lifecycle: firmware, telemetry, events, composability, even liquid-cooling CDUs
DiscoverabilityNone — you memorize commandsHypermedia: every response links to related resources
Practical note: most BMCs today speak both. Security guidance is generally to use Redfish and disable IPMI-over-LAN where you can.

Things worth knowing before you touch one

  • The BMC is a crown-jewel attack target. It sits below the OS with power over everything, so a compromised BMC is close to game over. Standard hygiene: put BMCs on an isolated management VLAN, never expose them to the internet, change default credentials immediately, keep BMC firmware patched, and disable IPMI-over-LAN if you don't need it.
  • Vendors extend the standard. The core tree is portable, but you'll meet "Oem": { "Dell": … } sections everywhere. Common operations are vendor-neutral; the exotic ones often aren't. Check each vendor's Redfish docs (and their interoperability profile support).
  • Use sessions, not basic auth, for real work. POST to SessionService/Sessions, get an X-Auth-Token, and use that — it's faster and avoids hammering the BMC's (slow, tiny) processor with repeated logins.
  • BMCs are small computers — treat them gently. A BMC is far less powerful than its host. Aggressive polling can slow or wedge it. For monitoring at scale, prefer EventService subscriptions (push) over tight GET loops (pull).
  • Expect self-signed certificates out of the box. Plan for certificate management (or explicit trust decisions) in any automation; blindly disabling TLS verification fleet-wide is a common bad habit.
  • Desktops don't have one. BMCs live in servers and workstation-class boards. The laptop analogue of "management below the OS" is different tech (e.g. Intel AMT/vPro) — related idea, different standard.
  • You can learn all of this with zero hardware. Section 9 is a step-by-step lab using DMTF's fake BMC. Start there before touching production gear.

Run it yourself — the zero-hardware lab

The fastest way to build intuition is to poke a live resource tree. DMTF ships a mockup server: a fake BMC that serves a realistic Redfish tree from JSON files on your laptop. Ten minutes, no server required.

LAB A — Curl a fake BMC

~10 min · needs Docker (or Python 3 + git)

  1. Start the mockup server. The official image bundles a sample rack-mount server tree:

    docker run --rm -p 8000:8000 dmtf/redfish-mockup-server

    No Docker? Clone DMTF/Redfish-Mockup-Server and run it with Python; sample trees ("mockup bundles") are downloadable as DSP2043.

  2. Hit the service root — the front door every Redfish service must expose:

    curl -s http://localhost:8000/redfish/v1/ | jq

    EXPECT: JSON listing Systems, Chassis, Managers — the tree from FIG. 4-1, live.

  3. Walk to a server and read its vitals. Follow the @odata.id links in each response (that's the OData hypermedia at work):

    curl -s http://localhost:8000/redfish/v1/Systems | jq
curl -s http://localhost:8000/redfish/v1/Systems/<id-from-above> \
  | jq '{Model, PowerState, Status}'

    EXPECT: model name, power state, health — the INSPECT tab of Section 4, reproduced by you.

  4. Explore on your own. Try /Chassis/…/Thermal for fan and temperature sensors, /Managers to meet the (fake) BMC itself, and /redfish/v1/$metadata to see the schema machinery. When JSON gets tedious, install DMTF's CLI and let it do the walking:

    pip install redfishtool
redfishtool -r localhost:8000 Systems list

LAB B — Boot a real BMC (in an emulator)

~1 hr · needs QEMU

  1. Run OpenBMC under QEMU. The mockup serves canned JSON; this runs the actual firmware — a full embedded Linux with the bmcweb Redfish server — emulating a real board (e.g. "romulus"). Follow the OpenBMC dev-environment guide to grab a prebuilt image and launch it.

  2. Log in and curl it over HTTPS — now with real authentication: create a session at /redfish/v1/SessionService/Sessions, take the X-Auth-Token, and repeat Lab A's walk against a BMC that's genuinely computing its answers.

LAB C — Graduate to real metal

optional · used servers are cheap

  1. Get one machine with a BMC. Off-lease Dell R-series / HPE DL-series boxes go for little on the used market, and most homelab-tier server boards (Supermicro, ASRock Rack) include one. Plug the dedicated mgmt port into your network, find its IP from your router or the boot screen.

  2. Change the default password first (Section 8, item 01 — this is not optional), keep it off the internet, then point everything you practiced in Labs A–B at https://<bmc-ip>/redfish/v1/. Power the host off and watch the BMC keep answering — the standby-domain trick from FIG. 3-1, in person.

Glossary — the jargon, decoded

DMTF
The standards body behind Redfish (and SMBIOS). Formerly "Distributed Management Task Force." dmtf.org
SoC — System on Chip
A whole computer (CPU, RAM controller, peripherals) on one chip. The BMC is a small ARM SoC.
Standby power rail
A low-power supply line (e.g. 5V-standby) that stays energized whenever the PSU has AC — even with the server "off." It's what keeps the BMC alive.
TelemetryService
The Redfish service for metrics at scale: define reports of sensor readings (temps, power) and collect them in bulk or on a schedule, instead of GETting each sensor one by one. details in DSP2046
OData
REST conventions Redfish borrows — the reason for properties like @odata.id (a resource's own URL, used as a link). odata.org
KVM-over-IP
Keyboard, Video, Mouse over the network: the BMC captures the host's video output and injects keystrokes, so you get a full remote console — BIOS screens included.
SOL — Serial-over-LAN
The host's serial console redirected over the network by the BMC. The text-mode cousin of KVM; ideal for kernel consoles.
SEL — System Event Log
The BMC's own hardware log (ECC errors, thermal trips, PSU faults). Persists across reboots and OS reinstalls.
Virtual media
The BMC presents a remote ISO/disk image to the host as a USB device. The host can boot from it — remote OS installs with no physical media.
PXE
"Preboot eXecution Environment" — booting an OS installer over the network from the NIC, before any disk is touched. The classic provisioning path. wikipedia
I2C / SMBus / PMBus
Slow two-wire buses connecting the BMC to temperature sensors, fan controllers, DIMM ID chips, and (as PMBus) power supplies. wikipedia
eSPI / LPC
The chipset-level link between BMC and host CPU — used for host console traffic, POST codes, and legacy IPMI-style host communication. eSPI is the modern replacement for the older LPC bus. wikipedia
NC-SI
"Network Controller Sideband Interface" — lets the BMC share the host's regular NIC for management traffic instead of needing its own dedicated port. wikipedia
OEM extensions
Vendor-specific additions inside "Oem": {…} sections of Redfish JSON. Legal per the spec, but not portable across vendors.
FRU
"Field-Replaceable Unit" — any part you can swap (PSU, fan, DIMM). FRU data = the ID/serial info the BMC reads from each part.
CDU / CoolingLoop
Coolant Distribution Unit and the Redfish schemas for liquid cooling (pumps, loops, leak detectors) — increasingly relevant in AI data centers.
